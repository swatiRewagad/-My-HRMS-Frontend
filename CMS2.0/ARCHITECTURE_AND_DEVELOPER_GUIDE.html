4. Functional Flow

4.1 Complaint Registration - End-to-End Flow

[Customer] ---> [Angular Portal] ---> [API Gateway] ---> [Eligibility Service] | | | (If eligible) | Drools rules v v [Complaint Form] ---> [API Gateway] ---> [Ingestion Service] | +---------+---------+ | | | v v v [Oracle DB] [History] [Outbox] | (Scheduled Poll)| v [Kafka: complaint.ingested] | +----------------+------------------+-----------------+ | | | | v v v v [Assignment Svc] [Workflow Svc] [Search Svc] [Notification Svc] | | | | | Drools | jBPM | OpenSearch | Email v v v v [complaint.assigned] [Start Process] [Index Doc] [Welcome Email]

4.2 Eligibility Check Flow

1
Customer visits portal landing page
2
Clicks "File a Complaint" — eligibility questionnaire appears
3
Questions fetched dynamically from QUESTION_MASTER table via REST API
4
Customer answers all mandatory questions
5
Answers posted to Eligibility Service for Drools rule evaluation
6
Drools Rules Execute:
• Court matter pending? → NOT ELIGIBLE
• Bank not approached? → NOT ELIGIBLE
• 30-day waiting period not met? → NOT ELIGIBLE
• Duplicate complaint? → NOT ELIGIBLE
• All checks passed? → ELIGIBLE
7
Result stored in ELIGIBILITY_AUDIT table (immutable record)
8
If ELIGIBLE → redirect to complaint registration form

4.3 Complaint Lifecycle (BPMN Process)

[START: Complaint Ingested] | v [Business Rule Task: Auto Assignment] --- Drools assigns team based on category | v [User Task: Officer Review & Accept] --- Human task, assigned to team | | | [Boundary Timer: 30 days SLA] | | v v [Exclusive Gateway] [Escalation Task] / \ | v v | [Under Review] [Escalation] | \ / | v v | [Resolution Gateway] <------+ | v [Service Task: Notify Customer] | v [Intermediate Timer: 7 days wait for confirmation] | v [END: Complaint Closed]

Scheduled Job runs every hour

5. Tech Stack Integration

5.1 Spring Boot + Oracle (JPA)

  • Connection: Oracle JDBC driver ojdbc11 version 23.4.0.24.05
  • Entities: JPA annotations with Oracle-specific sequences (@SequenceGenerator)
  • Repositories: Spring Data JPA (JpaRepository) for CRUD
  • Transactions: @Transactional on service methods
  • Schema: Managed via Flyway migration (database/V1__initial_schema.sql)

5.2 Transactional Outbox Pattern

Why Transactional Outbox?
Guarantees that every complaint event is published to Kafka exactly once, even if the application crashes between DB write and Kafka send.
1
IngestionService.registerComplaint() saves both ComplaintMaster AND OutboxEvent in a single transaction
2
OutboxPublisherService runs on a fixed schedule (every 5 seconds)
3
@SchedulerLock (ShedLock + SHEDLOCK table) prevents multiple instances from polling simultaneously
4
Publisher reads PENDING events, sends to Kafka, marks as PUBLISHED
5
Failed events increment retryCount; after 5 retries, marked as FAILED

5.3 Drools Rule Engine Integration

Configuration: KieContainer bean created from classpath DRL files

Rule FilePurposeRules Count
eligibility-rules.drlComplaint eligibility pre-check5 rules
assignment-rules.drlCategory-based team assignment7 rules
escalation-rules.drlSLA breach level determination4 rules
 
// Integration Pattern
KieSession kieSession = kieContainer.newKieSession();
kieSession.insert(fact);     // Insert fact object
kieSession.fireAllRules();   // Execute matching rules
kieSession.dispose();        // Clean up
// fact object now contains rule outputs (e.g., assignedTeam)

5.4 jBPM Workflow Engine

  • Process Definition: complaint-lifecycle.bpmn2 (BPMN2 XML)
  • Runtime: RuntimeManager creates KIE sessions per process instance
  • Tasks: BusinessRuleTask (Drools), UserTask (human), ServiceTask (code), Timer events
  • Integration: Kafka listener starts process on complaint.ingested event
  • SLA Timer: 30-day boundary timer triggers escalation if officer doesn't resolve

5.5 Keycloak OAuth2/OIDC

  • API Gateway acts as OAuth2 Resource Server
  • JWT tokens validated against Keycloak's JWKS endpoint
  • Custom JwtAuthenticationConverter reads realm_access.roles from JWT
  • Roles: cms_admin, cms_officer, cms_supervisor
  • Gateway routes enforce role-based access per service path

5.6 OpenSearch Full-Text Search

  • Indexing: Kafka consumer indexes every complaint event to cms-complaints index
  • Search: Multi-match query across subject, description, complainant name, entity name
  • Filter: Term queries for status-based filtering via FieldValue.of()
  • Transport: Apache HttpClient5 async NIO connection manager

5.7 Angular Frontend + Backend

  • Angular services call REST APIs via HttpClient
  • Environment-based API URL configuration (environment.ts / environment.prod.ts)
  • Standalone components with lazy-loaded routes
  • Angular Signals for reactive UI state

// In src/main/resources/rules/your-rules.drl
package com.rbi.cms.rules.your-domain;

import com.rbi.cms.common.dto.AssignmentFact;

rule "New Category - Assign to New Team"
    salience 100
    when
        $fact : AssignmentFact(category == "NEW_CATEGORY")
    then
        $fact.setAssignedTeam("NEW_TEAM");
        update($fact);
end

8. Build Fixes Applied

During initial project setup, the following compilation issues were identified and resolved to achieve a successful build of all 16 modules:

Fix 1: WorkflowService - Return Type Mismatch

Problem: ProcessInstance.getId() returns String in KIE API, but method declared Long return type.

File: cms-workflow-service/.../WorkflowService.java:41

// Before
public Long startComplaintWorkflow(ComplaintEvent event) { ... }

// After
public String startComplaintWorkflow(ComplaintEvent event) { ... }

Fix 2: AssignmentService - Cross-Module DTO Dependency

Problem: AssignmentFact was in cms-rules-service package but needed by cms-assignment-service. Cross-service dependency on a Spring Boot executable JAR is not supported.

Fix: Moved AssignmentFact and EscalationFact DTOs to cms-common module under com.rbi.cms.common.dto. Updated imports in assignment service and DRL files.

Fix 3: SearchService - Apache HttpClient5 Missing

Problem: OpenSearch Java Client 2.18.0 uses org.apache.hc.client5 classes but these weren't declared in the POM.

Fix: Added httpclient5, httpcore5, and httpcore5-h2 dependencies to cms-search-service/pom.xml.

Fix 4: SearchService - OpenSearch FieldValue API Change

Problem: OpenSearch Java Client 2.18 requires FieldValue.of(string) instead of passing string directly to .value().

// Before
.value(status)

// After
.value(FieldValue.of(status))

Fix 5: Infrastructure Module - Missing Spring Security Dependency

Problem: KeycloakJwtConfig uses Spring Security OAuth2 classes but POM didn't declare the dependency.

Fix: Added spring-boot-starter-oauth2-resource-server to cms-infrastructure/pom.xml.

Fix 6: Angular - TypeScript Params Type Error

Problem: Conditional params category ? { category } : {} created a union type incompatible with HttpClient's expected type.

// Before
const params = category ? { category } : {};

// After
const params: Record<string, string> = {};
if (category) { params['category'] = category; }

Fix 7: JAVA_HOME Configuration

Problem: Maven was using Java 17.0.18 instead of Java 21.

Fix: Set JAVA_HOME permanently to C:\Program Files\Microsoft\jdk-21.0.11.10-hotspot via system environment variable.

Build Result After Fixes
All 16 modules compile successfully. Angular frontend builds without errors. Total build time: ~51 seconds.

9. Sample Use Case with Flow

Use Case: Customer Files an ATM Cash Withdrawal Complaint

Scenario
A customer attempted to withdraw Rs. 15,00,000 from an ATM but the cash was not dispensed, while the amount was debited. They approached their bank 35 days ago but received no resolution.

Step 1: Customer Visits Portal

URL: http://localhost:4200

Angular LandingComponent renders hero section with "File a Complaint" and "Track Complaint" buttons.

Step 2: Eligibility Check

Component: EligibilityQuestionnaireComponent

API: GET /api/v1/eligibility/questions → 6 questions from DB

QuestionCustomer Answer
Court matter pending?NO
Approached bank?YES
30 days elapsed?YES
Duplicate complaint?NO
Category?ATM
Jurisdiction?MAHARASHTRA

API: POST /api/v1/eligibility/check

Drools Result: Rule "All Checks Passed" fires → ELIGIBLE

Step 3: Complaint Registration

Component: ComplaintFormComponent

API: POST /api/v1/complaints/register

Backend Processing (IngestionService):

1
Validation: Channel, category, required fields checked
2
ID Generation: ComplaintIdGenerator.generate() → CMS-2026-000147
3
Priority: Amount > 10 Lakh → Priority.HIGH
4
SLA: Instant.now() + 30 days = 2026-06-25
5
Save: COMPLAINT_MASTER record persisted to Oracle
6
History: Status null → NEW, action "COMPLAINT_REGISTERED"
7
Outbox: OUTBOX_EVENT record created (status=PENDING, topic=complaint.ingested)

Step 4: Outbox Publisher Polls & Publishes

Service: OutboxPublisherService (runs every 5 seconds)

  • @SchedulerLock acquires distributed lock in SHEDLOCK table
  • Finds PENDING event for CMS-2026-000147
  • kafkaTemplate.send("complaint.ingested", "CMS-2026-000147", payload)
  • On success: status → PUBLISHED

Step 5: Assignment Service Consumes Event

Listener: ComplaintIngestedAssignmentListener

Drools Execution (assignment-rules.drl):

  • Rule "ATM Complaints" fires → assignedTeam = "ATM_TEAM"
  • Rule "High Priority - Auto Escalate" fires → escalated = true, level = 1

Publishes complaint.assigned event to Kafka.

Step 6: Workflow Service Starts BPMN Process

  • jBPM creates process instance from complaint-lifecycle.bpmn2
  • Business Rule Task executes auto-assignment
  • User Task created: "Officer Review & Accept" assigned to ATM_TEAM
  • 30-day SLA boundary timer starts

Step 7: Search Service Indexes Document

Kafka consumer indexes all complaint fields to OpenSearch index cms-complaints.

Step 8: Notification Service Sends Email

Sends confirmation email: "Complaint Registered - CMS-2026-000147. Expected resolution by 2026-06-25."

Step 9: Officer Resolves (Day 5)

  • Officer logs in via Keycloak, claims task
  • Reviews complaint, contacts bank
  • Submits resolution: "Amount refunded to customer account"
  • BPMN: Under Review → Resolution Gateway → Notify Customer

Step 10: Auto-Close After 7 Days

  • BPMN intermediate timer waits 7 days for customer response
  • Timer expires → EndEvent → Status: CLOSED
  • complaint.closed event published

Final Database State

-- COMPLAINT_MASTER
COMPLAINT_ID = 'CMS-2026-000147'
STATUS       = 'CLOSED'
PRIORITY     = 'HIGH'
CATEGORY     = 'ATM'
ASSIGNED_TEAM = 'ATM_TEAM'
RESOLUTION_SUMMARY = 'Amount refunded to customer account'
RESOLVED_AT  = '2026-05-31'
CLOSED_AT    = '2026-06-07'

-- COMPLAINT_HISTORY (5 records)
1. null       → NEW        (COMPLAINT_REGISTERED)
2. NEW        → ASSIGNED   (AUTO_ASSIGNMENT)
3. ASSIGNED   → IN_PROGRESS (OFFICER_ACCEPTED)
4. IN_PROGRESS → RESOLVED   (RESOLUTION_SUBMITTED)
5. RESOLVED   → CLOSED     (AUTO_CLOSED_AFTER_7_DAYS)
