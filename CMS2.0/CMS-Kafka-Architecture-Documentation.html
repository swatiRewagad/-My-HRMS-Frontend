1. Overview - Role of Kafka in CMS

Apache Kafka acts as the central nervous system connecting all CMS microservices. When a complaint moves through its lifecycle, events are published to Kafka topics, and multiple services react independently.

Key Principle: Kafka decouples complaint processing. The backend just says "complaint arrived" and 4+ services independently do their jobs (assign, route workflow, index for search, notify complainant).

Why Kafka?

  • Decoupling: Services don't call each other directly
  • Fan-out: One event triggers multiple independent actions
  • Resilience: If a service is down, messages wait in the topic
  • Ordering: Per-complaint ordering guaranteed via partition keys
  • Scalability: Add more instances without changing code

4. Consumer Groups - How Services Know "It's For Me"

Key Insight: Events are NOT "for" one service. Every event goes to ALL subscribed services. Each consumer group gets its OWN independent copy of every message.

Consumer Groups in CMS

ServiceConsumer Group IDTopics Subscribed
Workflow Servicecms-workflow-groupcomplaint.ingested
Assignment Servicecms-assignment-groupcomplaint.ingested
Search Servicecms-search-groupALL topics (ingested, assigned, escalated, resolved, closed)
Notification Servicecms-notification-groupingested, assigned, resolved, escalated
Audit Servicecms-audit-groupALL topics

How Kafka Delivers Messages

complaint.ingested fires (1 message published) | | Kafka delivers a COPY to each consumer group | +---> cms-workflow-group -> Starts BPMN process [DELIVERED] +---> cms-assignment-group -> Runs Drools, assigns team [DELIVERED] +---> cms-search-group -> Indexes in OpenSearch [DELIVERED] +---> cms-notification-group -> Sends SMS/email ack [DELIVERED] All 4 services process the SAME event simultaneously. Nobody "claims" it.

Why a Service Does NOT Process an Event

Example: Why doesn't Assignment Service react to complaint.escalated?

Kafka Broker has topic: complaint.escalated | | "Who subscribed to me?" | +-- cms-notification-group -> YES (has @KafkaListener) -> DELIVER +-- cms-search-group -> YES (has @KafkaListener) -> DELIVER +-- cms-workflow-group -> NO (never subscribed) -> SKIP +-- cms-assignment-group -> NO (never subscribed) -> SKIP
Analogy: It's like a mailing list. You only get emails if you subscribed. The sender doesn't choose recipients.

7. Assignment Service & Drools Rules

What It Does

Listens to complaint.ingested, runs Drools business rules to decide which team/officer handles the complaint, then publishes complaint.assigned.

// AssignmentService.java
AssignmentFact fact = AssignmentFact.builder()
    .complaintId(complaintId)
    .category(category)        // e.g., "ATM", "LOAN", "UPI"
    .priority(priority)        // e.g., "HIGH", "MEDIUM"
    .amountInvolved(amount)
    .build();

KieSession kieSession = kieContainer.newKieSession();
kieSession.insert(fact);
kieSession.fireAllRules();     // Drools rules decide team

// After rules fire, fact is populated with:
// fact.assignedTeam = "RBIO_TEAM_A"
// fact.assignedOfficer = "officer name"
// fact.isEscalated = false

Then Publishes Second Kafka Event

// ComplaintIngestedAssignmentListener.java
ComplaintEvent assignedEvent = ComplaintEvent.builder()
    .complaintId(event.getComplaintId())
    .currentStatus(ComplaintStatus.ASSIGNED)
    .assignedTo(result.getAssignedTeam())
    .build();

kafkaTemplate.send(KafkaTopics.COMPLAINT_ASSIGNED,
    event.getComplaintId(), payload);
Summary: Assignment Service is a decision-maker. Takes complaint facts in -> applies Drools business rules -> outputs team assignment -> tells the rest of the system via Kafka.

8. How Complaints Reach an Officer's Screen

Key Insight: It's NOT push (Kafka -> browser). It's pull (officer logs in -> API query -> DB filter). The DB is the bridge between Kafka and the UI.

Step-by-Step

1. Kafka: complaint.ingested | v 2. Assignment Service (Drools rules): category=ATM, priority=HIGH -> team=CEPC, role=CEPC_OFFICER | v 3. DB Updated: complaint.department = "CEPC" complaint.assignedRole = "CEPC_OFFICER" complaint.assignedOfficer = "cepc.officer" complaint.status = "assigned" | v 4. Kafka: complaint.assigned (tells Search & Notification services) | v 5. Officer logs in via Keycloak SSO - Token contains roles: ["CEPC_OFFICER"] - Frontend detects department = CEPC | v 6. Frontend calls API: GET /api/v1/workflow/cepc/tasks?role=CEPC_OFFICER | v 7. Backend queries DB (WorkflowController.java): WHERE department="CEPC" AND assignedRole="CEPC_OFFICER" AND status NOT IN (resolved, closed...) | v 8. Returns only complaints assigned to that role/officer

The Bridge Pattern

Kafka (async, event-driven) DB (state store) UI (sync, request-driven) | Assignment Service --writes--> complaint table <--reads-- Officer's browser department="CEPC" assignedRole="CEPC_OFFICER" status="assigned"
Analogy: Manager puts file in your desk drawer (Assignment writes to DB). You come to office, open drawer, see files (Officer logs in, frontend queries API, sees assigned complaints).

9. Partitions & Ordering Guarantees

Topic Configuration

// KafkaTopicConfig.java
private static final int PARTITIONS = 6;
private static final int REPLICAS = 3;

Message Key = Complaint Number

kafkaTemplate.send(TOPIC, complaint.getComplaintNumber(), message);
//                        ^^^^^^^^ KEY ^^^^^^^^

Why 6 Partitions?

Allows up to 6 parallel consumers per group. If you run 3 instances of the workflow service (all with groupId = "cms-workflow-group"), Kafka assigns 2 partitions to each instance.

Why Complaint Number as Key?

All events for the same complaint always go to the same partition -> same consumer instance -> guarantees ordering per complaint. You'll never process "ASSIGNED" before "INGESTED" for the same complaint.

10. Topic Routing - How It's Decided

Answer: Topic routing is HARDCODED in each service's code, NOT configurable at runtime. There is no broker-side routing or dynamic configuration.

Each Service Always Publishes to the Same Topic

ServiceAlways Publishes ToDecided By
Ingestion Servicecomplaint.ingestedHardcoded constant
Assignment Servicecomplaint.assignedHardcoded constant
SLA Monitorcomplaint.escalatedHardcoded constant

Status-Based Switch (Only in IngestionService)

// IngestionService.java - for status transitions
String topic = switch (newStatus) {
    case RESOLVED   -> KafkaTopics.COMPLAINT_RESOLVED;
    case ESCALATED  -> KafkaTopics.COMPLAINT_ESCALATED;
    case CLOSED     -> KafkaTopics.COMPLAINT_CLOSED;
    case IN_PROGRESS -> KafkaTopics.COMPLAINT_IN_PROGRESS;
    case ASSIGNED   -> KafkaTopics.COMPLAINT_ASSIGNED;
    default        -> null;
};

Even this is a code-level mapping (compile-time), not an external configuration.

Similarly, Listeners are Hardcoded

// Each service's listener decides what to subscribe to at compile time
@KafkaListener(topics = KafkaTopics.COMPLAINT_INGESTED, groupId = "cms-workflow-group")
@KafkaListener(topics = KafkaTopics.COMPLAINT_INGESTED, groupId = "cms-assignment-group")
@KafkaListener(topics = KafkaTopics.COMPLAINT_ESCALATED, groupId = "cms-notification-group")

11. Outbox Pattern - Reliability

The cms-outbox-publisher service ensures events are published reliably even if Kafka is temporarily down.

How It Works

Normal Flow: DB Save + Kafka Publish (in same transaction? NO - dual write problem!) Outbox Pattern: 1. DB Save + Write event to OUTBOX table (same transaction - atomic) 2. Outbox Publisher polls OUTBOX table 3. Publishes to Kafka 4. Marks outbox entry as published If Kafka is down: - Outbox entries accumulate in DB - When Kafka recovers, publisher catches up - No events lost!
Guarantee: If the complaint is in the DB, the event WILL eventually reach Kafka. No data loss even during Kafka outages.

12. Configuration Reference

application.yml - Consumer Configuration

spring:
  kafka:
    bootstrap-servers: localhost:9092
    consumer:
      group-id: cms-workflow-group       # Unique per service
      auto-offset-reset: earliest        # Start from beginning if no offset
      enable-auto-commit: false          # Manual ack for at-least-once
    listener:
      ack-mode: MANUAL                   # Explicit acknowledgement
