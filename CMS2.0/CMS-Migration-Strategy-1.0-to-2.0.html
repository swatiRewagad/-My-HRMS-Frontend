2. Migration Strategy Options

Option A: Big Bang Migration

Replace CMS 1.0 entirely with CMS 2.0 in a single cutover event. All users switch on the same date.

Pros
  • Clean cutover, no dual-system maintenance
  • Simpler architecture during migration (no bridges)
  • Lower total project duration
  • No data sync complexity
Cons
  • Extremely high risk (all-or-nothing)
  • Extended downtime during cutover
  • No gradual learning curve for users
  • Difficult to rollback once data flows into new system
  • Requires complete feature parity before go-live

When to choose: Small user base, non-critical system, simple data model, or when CMS 1.0 has very few active users.

Option B: Strangler Fig Pattern (Incremental Migration)

Gradually route traffic from CMS 1.0 features to CMS 2.0 services, one module at a time. CMS 1.0 shrinks until it can be decommissioned.

Pros
  • Low risk - each module migrated independently
  • Continuous delivery of value
  • Easy rollback per module
  • Team learns microservices patterns gradually
  • Users adapt incrementally
  • Can pause/reprioritize mid-migration
Cons
  • Longer total duration
  • Dual-system maintenance during transition
  • Requires API gateway/routing layer
  • Data synchronization complexity
  • Integration testing across both systems

When to choose: Production system with active users, complex business logic, need for zero/minimal downtime, regulatory compliance requirements.

Option C: Parallel Run (Blue-Green)

Run both CMS 1.0 and CMS 2.0 simultaneously. New complaints go to 2.0, existing complaints continue in 1.0 until resolved.

Pros
  • Zero disruption to ongoing complaints
  • Natural drain of old system
  • Can compare outputs for validation
  • Rollback is simply routing back to 1.0
Cons
  • High infrastructure cost (double the resources)
  • Staff must learn and use both systems
  • Reporting/analytics split across two systems
  • Long tail of old complaints may extend parallel period

When to choose: Critical system where no data loss is acceptable, regulatory requirement for continuous processing, sufficient infrastructure budget.

Option D: Hybrid (Strangler + Parallel for Critical Modules)

Use Strangler Fig for most modules, but run critical modules (complaint processing, SLA tracking) in parallel for a validation period before full cutover.

Pros
  • Best of both worlds - fast for simple modules, safe for critical ones
  • Validation period builds confidence
  • Flexible timeline per module
Cons
  • Most complex to implement
  • Requires sophisticated routing and sync
  • Team needs expertise in multiple patterns

When to choose: Mix of critical and non-critical modules, experienced team, available infrastructure for parallel runs on select services.

3. Recommended Approach: Strangler Fig Pattern

Why Strangler Fig for CMS? CMS 2.0 already has 14+ microservices designed around bounded contexts (eligibility, ingestion, workflow, etc.). This natural decomposition aligns perfectly with incremental migration where each service boundary becomes a migration unit. The API Gateway in CMS 2.0 serves as the natural routing layer for the strangler pattern.

How It Works

The strangler fig pattern works by:

  1. Intercept: Place the CMS 2.0 API Gateway in front of CMS 1.0
  2. Route: Initially, all traffic passes through to CMS 1.0
  3. Migrate: One service at a time, implement in CMS 2.0 and redirect traffic
  4. Verify: Run shadow traffic or feature flags to validate
  5. Decommission: Remove the old code path from CMS 1.0

4. Phase-wise Migration Plan

Foundation & Infrastructure

4-6 Weeks
  • Deploy Keycloak and configure CMS realm with roles (CRPC, CEPC, Officer, Staff, Admin)
  • Set up Kafka cluster and topic structure
  • Deploy API Gateway with routing rules (initial pass-through to CMS 1.0)
  • Set up OpenSearch cluster
  • Configure Prometheus + Grafana monitoring
  • Set up Docker/OpenShift deployment pipelines
  • Implement CDC (Change Data Capture) from CMS 1.0 database for sync

Identity & Access Migration

2-3 Weeks
  • Migrate CmsUser records to Keycloak (scripted bulk import)
  • Map existing roles/permissions to Keycloak realm roles
  • Implement SSO bridge: CMS 1.0 accepts Keycloak tokens
  • Update CMS 1.0 frontend to authenticate via Keycloak
  • Validate: all CMS 1.0 features work with new auth

Peripheral Services Migration

4-5 Weeks
  • Deploy and route: Eligibility Service (new feature)
  • Deploy and route: File Storage Service (migrate attachment paths)
  • Deploy and route: Notification Service (new feature)
  • Deploy Search Service indexing from CMS 1.0 database
  • Feature flag: gradually enable new services for internal users first

Core Complaint Processing Migration

6-8 Weeks
  • Deploy Ingestion Service - new complaints flow to CMS 2.0
  • Deploy Rules Service with migrated CategorizationRules to Drools
  • Deploy Assignment Service with round-robin logic
  • Shadow mode: process complaints in both systems, compare results
  • Migrate historical complaint data (batch ETL)
  • Cutover: route all new complaint traffic to CMS 2.0

Workflow & Advanced Features

4-6 Weeks
  • Deploy Camunda Workflow Service
  • Model existing complaint states as BPMN processes
  • Deploy SLA Monitor Service
  • Deploy Audit Service (event sourcing from Kafka)
  • Migrate in-flight complaints: map current states to workflow states
  • Enable OCR/AI features for document processing

Frontend & UI Migration

4-5 Weeks
  • Deploy CMS 2.0 Portal Frontend (cms-portal-frontend)
  • Migrate public complaint filing form
  • Migrate officer/staff dashboards
  • Migrate admin panels
  • Enable role-specific views (CRPC, CEPC dashboards)
  • Redirect all traffic from CMS 1.0 frontend to CMS 2.0 portal

Decommission & Cleanup

2-3 Weeks
  • Verify zero traffic to CMS 1.0
  • Archive CMS 1.0 database (read-only backup)
  • Remove API Gateway routes to CMS 1.0
  • Decommission CMS 1.0 infrastructure
  • Update documentation and runbooks
  • Post-migration performance benchmarking

Data Migration Approaches

A. ETL (Extract-Transform-Load) - For Historical Data

When to use: Bulk migration of historical complaints, users, and reference data during a maintenance window.
  • Extract: Read from CMS 1.0 MySQL (complaints, users, categories, timelines)
  • Transform: Map field names, generate new IDs (sequences), enrich data (add jurisdiction, SLA dates)
  • Load: Insert into CMS 2.0 Oracle/MySQL with validation
  • Verify: Record counts, checksum comparison, sample audit

B. Change Data Capture (CDC) - For Live Sync

When to use: During the parallel run period when both systems must reflect the same data.
  • Use Debezium to capture MySQL binlog changes from CMS 1.0
  • Publish changes to Kafka topics
  • CMS 2.0 services consume and apply changes
  • Handles real-time sync during transition period

C. Dual-Write - For Critical Data

Caution: Only use for the brief transition period of a specific module. Dual-write introduces consistency risks.
  • CMS 1.0 writes to both old DB and publishes to Kafka
  • CMS 2.0 services pick up from Kafka
  • Use outbox pattern to ensure atomicity
  • Reconciliation job runs periodically to detect drift

Data Integrity Safeguards

  • Maintain an ID mapping table (old_id → new_id) for all migrated entities
  • Run checksum validation after each batch migration
  • Keep CMS 1.0 database in read-only mode post-cutover for 30 days
  • Implement idempotent processing to handle duplicate events
  • Log every transformation for auditability

  • Deprecation warnings in response headers for v1 endpoints

7. Risk Assessment & Mitigation

Data Loss During Migration

Impact: Critical | Probability: Medium

Mitigation: Full backup before each phase, CDC for real-time sync, reconciliation jobs, 30-day read-only archive.

In-flight Complaint State Corruption

Impact: Critical | Probability: Medium

Mitigation: Freeze in-flight complaints during workflow migration, map states explicitly, validate each complaint post-migration.

Authentication Disruption

Impact: High | Probability: Low

Mitigation: Keycloak migration first, test with shadow users, maintain fallback to basic auth for 2 weeks.

Performance Degradation (Microservices Overhead)

Impact: Medium | Probability: Medium

Mitigation: Load testing before each phase, circuit breakers, connection pooling, caching strategy.

Kafka Message Ordering Issues

Impact: Medium | Probability: Medium

Mitigation: Partition by complaint ID, idempotent consumers, exactly-once semantics where possible.

Team Skill Gap (Microservices, Kafka, Keycloak)

Impact: Medium | Probability: High

Mitigation: Training before Phase 1, pair programming, dedicated DevOps support, runbook documentation.

Third-party Service Unavailability

Impact: Low | Probability: Low

Mitigation: Fallback OCR providers configured, circuit breaker patterns, graceful degradation.

SLA Breach During Migration

Impact: High | Probability: Low

Mitigation: Migrate during low-traffic windows, maintain SLA tracking in both systems during transition, alert on near-breach complaints.

8. Rollback Strategy

Golden Rule: Every phase must have a documented, tested rollback procedure that can be executed within 30 minutes.

Per-Phase Rollback Procedures

Phase Rollback Trigger Rollback Action Time to Rollback
Auth Migration Login failures > 5% or complete auth outage Switch API Gateway to bypass Keycloak, restore direct auth 5 minutes
Peripheral Services Service errors > 1% or latency > 3x baseline Route traffic back to CMS 1.0 endpoints via gateway config 2 minutes
Core Processing Data inconsistency detected or complaint loss Stop CMS 2.0 ingestion, re-enable CMS 1.0 intake, replay missed events 15 minutes
Workflow Migration Workflow state corruption or SLA miscalculation Freeze affected complaints, route new work to CMS 1.0, manual state repair 30 minutes
Frontend Migration Critical UI bugs blocking operations DNS/routing switch back to CMS 1.0 frontend 2 minutes

Rollback Prerequisites

  • CMS 1.0 infrastructure remains running until Phase 7 complete + 30 days
  • Database backups taken before and after each phase
  • API Gateway routing changes are config-only (no deployment needed)
  • Monitoring alerts configured for all rollback triggers
  • Rollback runbook reviewed and rehearsed before each phase

Critical Test Scenarios

  • File a new complaint through CMS 2.0 → verify it appears in dashboards, gets assigned, SLA set
  • Migrate a complaint mid-workflow → verify state continuity in CMS 2.0 workflow engine
  • User logs in via Keycloak → verify correct role-based access in both frontend and APIs
  • Kill a CMS 2.0 service → verify circuit breaker activates, no data loss, auto-recovery
  • High-volume complaint batch (1000+ in 1 minute) → verify no message loss, correct ordering
  • Rollback from Phase 4 → verify all new complaints since cutover are recoverable

10. Pre-Migration Checklist

Technical Readiness

  • All CMS 2.0 services passing CI/CD pipeline with >80% test coverage
  • API Gateway deployed and routing to CMS 1.0 (pass-through mode validated)
  • Keycloak realm configured with all roles and tested with sample users
  • Kafka cluster deployed, topics created, producer/consumer tested
  • OpenSearch cluster deployed and indexing working
  • Monitoring dashboards (Prometheus/Grafana) showing all service metrics
  • Docker/OpenShift manifests validated in staging environment
  • Database migration scripts tested against production-volume data
  • CDC (Debezium) pipeline tested with CMS 1.0 database
  • Load tests completed showing CMS 2.0 handles peak CMS 1.0 traffic + 50% headroom

Operational Readiness

  • Runbooks documented for all services (startup, shutdown, troubleshooting)
  • Alert rules configured for all critical paths
  • On-call rotation assigned for migration period
  • Rollback procedures documented and rehearsed
  • Communication plan for users (downtime windows, new features, training)
  • Backup and recovery procedures validated
  • Security review completed (OWASP, penetration testing)
  • Compliance sign-off obtained (data handling, audit trail)

Data Readiness

  • Complete data inventory of CMS 1.0 (row counts, sizes, relationships)
  • Data mapping document reviewed and approved
  • ID mapping strategy defined and tested
  • Data quality issues in CMS 1.0 identified and cleaned
  • Migration scripts tested with full production copy
  • Estimated migration duration calculated and within maintenance window
  • Verification queries prepared for post-migration validation

Organizational Readiness

  • Stakeholder sign-off on migration timeline
  • User training scheduled and materials prepared
  • Support team trained on CMS 2.0 troubleshooting
  • Go/No-Go criteria defined for each phase
  • Escalation path defined for migration issues

11. Decision Framework

When to Choose Which Strategy

If your situation is... Choose... Because...
< 100 active users, non-critical, can afford 4-8 hrs downtime Big Bang Fastest, simplest, lowest total cost
Production system, active complaints, regulatory compliance Strangler Fig Lowest risk, continuous delivery, easy rollback
Zero tolerance for any complaint processing gap Parallel Run Both systems handle complaints until full confidence
Mix of critical and non-critical features, large team Hybrid Optimizes for both speed and safety per module

Key Decision Points During Migration

Decision Options Criteria
Database: Keep MySQL or move to Oracle? MySQL (dev parity) vs Oracle (prod target) If production is Oracle, migrate to Oracle. Use MySQL only for dev/test. CMS 2.0 schema already supports Oracle sequences.
Migrate historical complaints or start fresh? Full migration vs archive-only vs partial Regulatory: if complaints must be accessible for N years, migrate. If only for reference, read-only archive is sufficient.
In-flight complaints: freeze or live-migrate? Freeze during window vs map states on-the-fly If SLA allows a maintenance window, freeze is safer. Otherwise, implement state mapping with manual verification.
Frontend: gradual migration or full rewrite? Module-by-module vs deploy new portal entirely CMS 2.0 portal (cms-portal-frontend) is already a complete rewrite with new features. Deploy entirely, train users once.
OCR rules: migrate existing or re-create? Auto-convert CategorizationRules to Drools vs manual rule design If rules are simple, auto-convert. If complex or business wants to redesign, take the opportunity to optimize.

Go/No-Go Criteria Per Phase

A phase should NOT proceed if any of these are true:
  • Previous phase has unresolved critical bugs
  • Performance benchmarks show >20% degradation vs baseline
  • Data reconciliation shows any discrepancy > 0 records
  • Rollback procedure has not been tested in staging
  • Team has not completed required training for the phase
  • Monitoring/alerting not confirmed operational

Summary & Next Steps

Key Takeaway: The Strangler Fig pattern is the recommended approach for CMS 1.0 → 2.0 migration. CMS 2.0's microservices architecture with an API Gateway naturally supports this pattern. Start with authentication (Keycloak), work outward through peripheral services, then tackle core complaint processing last.

Immediate Next Steps

  1. Inventory: Document exact record counts, active users, and in-flight complaints in CMS 1.0
  2. Stakeholder Alignment: Present this strategy, agree on approach and timeline
  3. Environment Setup: Provision staging environment matching CMS 2.0 architecture
  4. POC: Run a proof-of-concept for Phase 1 (API Gateway + Keycloak) in staging
  5. Team Training: Begin Kafka, Keycloak, and Docker/Kubernetes training
