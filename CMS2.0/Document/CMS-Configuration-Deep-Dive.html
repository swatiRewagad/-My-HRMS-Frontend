Java // ComplaintIngestedListener.java — THE TRIGGER @KafkaListener(topics = KafkaTopics.COMPLAINT_INGESTED, groupId = "cms-workflow-group") public void onComplaintIngested(String message, Acknowledgment ack) { ComplaintEvent event = objectMapper.readValue(message, ComplaintEvent.class); workflowService.startComplaintWorkflow(event); // ← Starts the process ack.acknowledge(); // ← Tells Kafka "processed" } // WorkflowService.java — STARTS THE BPMN PROCESS public String startComplaintWorkflow(ComplaintEvent event) { Map<String, Object> params = new HashMap<>(); params.put("complaintId", event.getComplaintId()); params.put("category", event.getPayload()); params.put("correlationId", event.getCorrelationId()); // Process ID must match: "com.rbi.cms.complaint-lifecycle" ProcessInstance pi = kieSession.startProcess(PROCESS_ID, params); return pi.getId(); }

Java // AssignmentService.java — HOW DROOLS EXECUTES public AssignmentFact assignComplaint(String complaintId, String category, String priority, Double amount) { AssignmentFact fact = AssignmentFact.builder() .complaintId(complaintId) .category(category) // "ATM" .priority(priority) // "HIGH" .amountInvolved(amount) .build(); KieSession kieSession = kieContainer.newKieSession(); try { kieSession.insert(fact); // Put fact into rule engine kieSession.fireAllRules(); // Execute all matching rules } finally { kieSession.dispose(); } // fact.assignedTeam = "ATM_TEAM", fact.escalated = true return fact; }
Drools (.drl) rule "ATM Complaints - Assign to ATM Team" salience 100 // fires FIRST (highest priority) when $fact : AssignmentFact(category == "ATM") // CONDITION: category matches then $fact.setAssignedTeam("ATM_TEAM"); // ACTION: set team update($fact); end rule "High Priority - Auto Escalate to Senior Officer" salience 50 // fires SECOND when $fact : AssignmentFact(priority == "HIGH" || priority == "CRITICAL") then $fact.setEscalated(true); // ACTION: mark escalated $fact.setEscalationLevel(1); update($fact); end

t = 5 days Officer Submits Resolution

BPMN gateway decides: normal resolution OR escalation path

BPMN XML (Gateway Logic) <!-- After officer submits, gateway decides the path --> <exclusiveGateway id="InvestigationGateway"/> <!-- Path 1: Normal resolution --> <sequenceFlow> <conditionExpression>#{escalationReason == null}</conditionExpression> <!-- No escalation → go to Resolution --> </sequenceFlow> <!-- Path 2: Needs escalation --> <sequenceFlow> <conditionExpression>#{escalationReason != null}</conditionExpression> <!-- Escalation reason exists → Escalated Review --> </sequenceFlow>

BPMN Conditions vs Rules — Key Difference

  • BPMN gateway conditions = simple routing ("which path to take") — rarely changes
  • Rules = complex business decisions ("which team, what level") — changes frequently
t = 12 days Auto-Close Timer Fires → Complaint CLOSED

7-day timer expires after resolution → workflow auto-closes → process instance ends

BPMN XML <!-- Timer waits 7 days after notification --> <intermediateCatchEvent id="ClosureTimer"> <timerEventDefinition> <timeDuration>P7D</timeDuration> <!-- ISO 8601: 7 days --> </timerEventDefinition> </intermediateCatchEvent> <!-- After timer fires → END --> <endEvent id="EndEvent" name="Complaint Closed"/> <!-- Process instance ends. Lifecycle complete. -->

Configuration Files Master Index

KAFKA EVENT FLOW:
├── Topic names → cms-common/.../KafkaTopics.java
├── Who publishes → IngestionService.insertOutboxEvent()
├── Who consumes → *Listener.java classes (@KafkaListener)
├── Consumer groups → application-dev-local.yml (each service)
└── Broker address → application-dev-local.yml (bootstrap-servers)
WORKFLOW (BPMN):
├── Process definition → processes/complaint-lifecycle.bpmn2
├── Process ID → "com.rbi.cms.complaint-lifecycle"
├── Timers (SLA/close) → <timeDuration> in .bpmn2
├── Gateway conditions → <conditionExpression> in .bpmn2
├── Task ownership → <potentialOwner> in .bpmn2
└── Which rules to call → tns:ruleFlowGroup attribute in .bpmn2
RULES (DROOLS):
├── Rule definitions → Database (managed via Admin UI)
├── Static rules (dev) → src/main/resources/rules/*.drl
├── Drools config → DroolsConfig.java (loads rules)
├── Fact classes → cms-common/.../dto/AssignmentFact.java
└── Engine reload → Rules Service /api/v1/rules/deploy
ROUTING & INFRASTRUCTURE:
├── API Gateway routes → cms-api-gateway/application.yml
├── Service ports → each service's application.yml
├── CORS config → DevLocalSecurityConfig.java
└── Database config → application-dev-local.yml per service

Adding a Completely NEW Workflow (e.g., "Loan Complaint Workflow")

Create new BPMN file

cms-workflow-service/src/main/resources/processes/loan-complaint-lifecycle.bpmn2
Process ID: "com.rbi.cms.loan-complaint-lifecycle"

Modify Kafka listener to route to correct workflow

In ComplaintIngestedListener.java: check category → start appropriate process

Create rules specific to the new workflow

Admin UI → Create new rules with category "LOAN_ASSIGNMENT"

Deploy

Restart workflow service (picks up new .bpmn2) + Deploy rules via Admin UI (no restart)
