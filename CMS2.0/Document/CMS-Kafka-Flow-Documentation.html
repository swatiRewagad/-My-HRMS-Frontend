Stage 2: Assignment ASSIGNED

2026-05-27 15:20:05 IST (auto, ~5 seconds after filing)

Event Consumption

Workflow Service (cms-workflow-group) consumes complaint.ingested The workflow service picks up the event and determines the appropriate team based on the complaint category.

Processing Steps

  1. Workflow service consumes complaint.ingested event
  2. Determines team using category mapping: ATM → ATM_TEAM
  3. Registers task in in-memory workflow store (task state: ASSIGNED)
  4. Calls back ingestion service via REST:
    REST Callback POST /api/v1/complaints/CMP-20260527-000001/assignment?team=ATM_TEAM
  5. Ingestion service updates DB: status=ASSIGNED, assigned_team=ATM_TEAM
  6. History record added: "Complaint assigned to ATM_TEAM"
  7. Kafka event published to complaint.assigned

ComplaintEvent Structure

All Kafka events follow a consistent ComplaintEvent schema. The payload field contains a JSON-serialized string with context-specific data.

ComplaintEvent.java // Java class definition public class ComplaintEvent { private String eventId; // UUID - unique event identifier private String complaintId; // CMP-YYYYMMDD-NNNNNN format private String previousStatus; // Status before transition (null for NEW) private String currentStatus; // Current status after transition private String payload; // JSON string with event-specific data private String correlationId; // UUID - tracks complaint across services private Instant occurredAt; // ISO-8601 timestamp of event }

Accessing Kafdrop

URL http://localhost:9000 // Kafdrop provides a web UI to inspect Kafka topics, messages, and consumer groups. // It runs as a Docker container alongside Kafka (see docker-compose.yml in cms-infra).

Browsing Topics

  1. Navigate to http://localhost:9000
  2. The home page shows all topics in the cluster
  3. Click on any topic name (e.g., complaint.ingested) to view details
  4. You will see partition count, replication factor, and message count
[Topic List View]

Kafdrop Home - Shows all complaint.* topics with partition info and message counts

Topics: complaint.ingested (3), complaint.assigned (3), complaint.inprogress (3), complaint.escalated (3), complaint.resolved (3), complaint.closed (3), complaint.dlq (1)

Viewing Messages

  1. Click on a topic name
  2. Click "View Messages" button
  3. Select partition (0, 1, or 2) or "All Partitions"
  4. Set offset: "Beginning" to see all, or specific offset number
  5. Set count: Number of messages to retrieve (default: 100)
  6. Click "View Messages" - messages appear with key, value, timestamp, and headers
[Message View]

Kafdrop Message View - Shows individual ComplaintEvent JSON payloads

Key: CMP-20260527-000001 | Partition: 1 | Offset: 0 | Timestamp: 2026-05-27T15:20:01Z

Tip: Message Key Messages are keyed by complaintId (e.g., CMP-20260527-000001). This ensures all events for the same complaint go to the same partition, maintaining ordering per complaint.

Checking Consumer Lag

  1. Click "Consumers" in the top navigation
  2. Find your consumer group (e.g., cms-workflow-group)
  3. Click to see per-partition details:
    • Current Offset: Last message processed by consumer
    • End Offset: Latest message in partition
    • Lag: Difference (messages waiting to be processed)
  4. If lag > 0 consistently, consumer may be stuck or slow

Inspecting Dead Letter Queue

  1. Navigate to topic complaint.dlq
  2. View Messages from beginning
  3. DLQ messages include headers with:
    • x-original-topic: Original topic that failed
    • x-exception-message: Error that caused failure
    • x-retry-count: Number of retries attempted
  4. Fix the root cause, then replay from DLQ
DLQ Alert If messages consistently appear in the DLQ, check service logs for the consuming service. Common causes: deserialization errors, database connectivity issues, or validation failures in downstream processing.

8 Troubleshooting

▼
PATCH Method Not Working on Windows
Symptom: HTTP 405 Method Not Allowed or connection reset when using PATCH endpoints.

Cause: Some Windows HTTP client libraries and proxies do not support the PATCH method properly.

Solution: The CMS system uses POST endpoints instead of PATCH for all status updates:
  • Use POST /api/v1/complaints/{id}/status?status=IN_PROGRESS instead of PATCH
  • Use POST /api/v1/complaints/{id}/assignment?team=ATM_TEAM instead of PATCH
  • Use POST /api/v1/workflow/{id}/transition?targetStatus=RESOLVED for workflow transitions
Consumer Not Receiving Messages
Symptom: Kafka events are published but consumer service does not process them.

Checklist:
  1. Topic name mismatch: Verify the topic name in producer matches consumer subscription. Common mistake: complaint.in-progress vs complaint.inprogress
  2. Group ID mismatch: Check spring.kafka.consumer.group-id in application.yml matches expected group:
    • Workflow: cms-workflow-group
    • Notification: cms-notification-group
    • Assignment: cms-assignment-group
  3. Kafka not running: Verify Kafka is up: docker ps | grep kafka
  4. Consumer started after producer: Set auto-offset-reset: earliest to read from beginning
  5. Deserialization error: Check logs for SerializationException. Ensure JsonDeserializer trusted packages include your event class package

Debug with Kafdrop: Check consumer lag at http://localhost:9000 > Consumers. If lag is growing, the consumer is not keeping up or is stuck.
H2 Data Lost on Restart
Symptom: After restarting the Ingestion Service, all complaints are gone.

Cause: H2 is configured in-memory mode (jdbc:h2:mem:cmsdb). Data exists only while the JVM is running.

Solutions:
  • For development: Simply re-file the complaint. Use the provided curl/Postman collection to quickly recreate test data.
  • For persistence (dev): Switch to file-based H2:
    application.yml spring.datasource.url: jdbc:h2:file:./data/cmsdb
  • For production: Use MySQL/PostgreSQL with proper persistence

Note: Kafka messages are persisted independently. Even if DB data is lost, Kafka retains events for the configured retention period (7 days default). You can replay events to rebuild state.
Messages in Dead Letter Queue (DLQ)
Symptom: Messages appear in complaint.dlq topic instead of being processed.

Diagnosis Steps:
  1. Open Kafdrop: http://localhost:9000 > topic complaint.dlq
  2. View messages - check the message headers for error details:
    • x-original-topic: Which topic the message came from
    • x-exception-message: The actual error
    • x-exception-stacktrace: Full stack trace
  3. Check consuming service logs for detailed error information

Common Causes:
  • Invalid complaint ID: Complaint referenced in event does not exist in DB
  • Invalid state transition: e.g., trying to move from NEW directly to RESOLVED
  • Service dependency down: Workflow service cannot reach Ingestion Service
  • Payload format change: Producer changed event format but consumer has old schema

Recovery: Fix the root cause, then use a DLQ replayer utility or manually re-publish the corrected event to the original topic.
Workflow Service Returns 404 for Task
Symptom: GET /api/v1/workflow/tasks?team=ATM_TEAM returns empty list even though complaint was filed.

Cause: Workflow service stores tasks in-memory. If the service was restarted after the complaint was filed, the task is lost.

Solution:
  1. Re-file the complaint (POST /api/v1/complaints)
  2. The ingestion service will publish to complaint.ingested
  3. Workflow service will consume the event and register the task
Kafka Connection Refused
Symptom: org.apache.kafka.common.errors.TimeoutException: Failed to update metadata

Checklist:
  1. Ensure Docker is running: docker ps
  2. Check Kafka container: docker ps | grep kafka
  3. Verify port 9092 is accessible: curl localhost:9092 (should get binary response, not connection refused)
  4. Restart infrastructure: cd cms-infra && docker-compose down && docker-compose up -d
  5. Wait 10-15 seconds for Kafka to fully initialize before starting services
