2. Key Challenges: CRM Next to Custom Platform

This is NOT a typical version upgrade. This is a platform replacement. Migrating from a COTS product (CRM Next) to a custom-built microservices platform is fundamentally different from upgrading within the same technology. Every layer needs to be rebuilt, not just upgraded.

Unique Challenges of CRM Next Migration

Challenge Why It's Hard Impact
Proprietary Data Schema CRM Next uses its own internal data model. Extracting data requires understanding CRM Next's metadata, relationships, and custom fields. No direct SQL mapping exists. CRITICAL
7 TB Data Volume At this scale, ETL takes days/weeks. Network bandwidth, storage provisioning, and validation become major projects themselves. CRITICAL
Business Logic Embedded in Config CRM Next's workflow rules, validations, and automations are stored as configuration, not code. These must be reverse-engineered and re-implemented in Camunda/Drools. HIGH
No Source Code Access CRM Next is a black box. Internal processing logic, event sequences, and data transformations are not visible. Must rely on API outputs and documentation. HIGH
User Training Gap Users trained on CRM Next's UI must learn an entirely new interface. This is not a minor version change - it's a completely different system. MEDIUM
Integration Re-wiring All external systems integrating with CRM Next APIs need to be re-pointed to CMS 2.0 API Gateway. Each integration must be mapped and tested. HIGH
Reporting Continuity CRM Next's built-in reports and dashboards must be recreated in CMS 2.0. Historical report data must remain accessible. MEDIUM
Compliance & Audit Trail Regulatory requirements demand continuity of audit trails. CRM Next's audit logs must be preserved and accessible from CMS 2.0. HIGH
Vendor Cooperation Needed Data export from CRM Next may require vendor support, export APIs, or professional services. Vendor may not be incentivized to help with migration away from their product. HIGH
Downtime Window Constraints With 7 TB of data, achieving zero-downtime migration is extremely difficult. Need to negotiate acceptable downtime windows with stakeholders. MEDIUM

What Makes This Different from a Typical Migration

Typical Upgrade (Same Platform)
  • Same data schema, just add columns
  • Same UI patterns, just reskin
  • Same APIs, just version bump
  • Vendor provides upgrade scripts
  • In-place upgrade possible
Platform Replacement (Our Case)
  • Complete data schema redesign
  • New UI from scratch (Angular)
  • All-new API contracts
  • Custom ETL scripts needed
  • Parallel systems required
  • Business logic re-implementation

3. The 7 TB Data Migration Challenge

7 TB is NOT a trivial data migration. At 1 Gbps network speed, transferring 7 TB takes ~15.5 hours of raw transfer time alone. With transformation, validation, and retries, the realistic timeline is 3-7 days for a full migration. This single factor dominates the entire migration strategy.

Attachments & Documents ~3-4 TB MEDIUM Background copy with lazy migration

Key Decisions for 7 TB Migration

Critical Questions to Answer Before Planning:
  • What data export APIs does CRM Next provide? Bulk export? Streaming? Rate limits?
  • Can we get direct database access? Read replica? Database dump?
  • What is the regulatory retention requirement? Must ALL 7 TB be migrated, or only active/recent data?
  • What downtime window is acceptable? This determines if we do big-bang data load or incremental
  • Is the data clean? Orphaned records, duplicates, corrupted entries?
  • What is the attachment storage format? File system? Object store? Database BLOBs?

4. Migration Strategy Options

Option A: Big Bang Migration

Complete data extraction from CRM Next, full transformation, and cutover to CMS 2.0 in a single maintenance window.

Pros
  • Clean cutover, single point in time
  • No dual-system maintenance
  • No data sync complexity
  • CRM Next license can be terminated immediately
Cons
  • 7 TB migration requires 3-7 day maintenance window
  • Extremely high risk - no fallback after cutover
  • All business logic must be perfectly replicated before go-live
  • Users face complete system change overnight
  • If anything fails, CRM Next must be restored (complex)

When to choose: NOT recommended for 7 TB data volume. Only viable if regulatory allows a week-long maintenance window and all data can be migrated in that time.

Option B: Phased Migration with Data Tiering

Migrate data in tiers: hot data (active complaints, recent records) first, then warm data (past 2 years), then cold data (archive). New complaints go to CMS 2.0 from a cutover date, while CRM Next remains read-only for historical lookups.

Pros
  • Manageable chunks - hot data is likely <500 GB
  • CMS 2.0 goes live quickly with active data
  • Cold/archive data migrates in background over weeks
  • CRM Next stays available for historical lookups
  • Risk is contained per tier
  • Rollback is straightforward (route back to CRM Next)
Cons
  • CRM Next license must continue during transition (cost)
  • Users may need to check two systems temporarily
  • Data consistency window during hot-data cutover
  • Longer total migration duration

When to choose: Best fit for 7 TB data volume. Minimizes downtime, allows gradual validation, and keeps CRM Next as safety net.

Option C: Parallel Run with Shadow Processing

Run both CRM Next and CMS 2.0 simultaneously. New complaints are processed in both systems. Compare outputs to validate CMS 2.0. Cut over when confidence is high.

Pros
  • Highest confidence in CMS 2.0 correctness
  • Zero disruption to operations
  • Validates business logic equivalence
  • Instant rollback (just stop using CMS 2.0)
Cons
  • Double infrastructure cost
  • Double CRM Next license fees during parallel
  • Complex to keep both systems in sync
  • Staff overhead (operating two systems)
  • Still need to migrate 7 TB eventually

When to choose: When regulatory requirements demand proof of equivalence before cutover, or when the political risk of failure is very high.

Option D: Strangler Fig + Archive Strategy

New complaints go to CMS 2.0 from day one. CRM Next becomes read-only archive. No bulk data migration - historical data accessed via CRM Next read-only instance or archived exports. Over time, as old complaints resolve, CRM Next data shrinks to irrelevance.

Pros
  • No 7 TB migration needed upfront
  • CMS 2.0 goes live immediately (for new work)
  • Natural attrition drains CRM Next
  • Lowest technical risk
  • Fastest time to value
Cons
  • CRM Next license continues indefinitely (read-only access)
  • Users must check two systems for historical data
  • Reporting split across systems
  • May not satisfy regulatory "single system of record" requirement
  • In-flight complaints must be handled (complete in old or migrate)

When to choose: When the 7 TB data is mostly archival, regulatory allows split access, and the priority is to stop CRM Next usage for new work ASAP.

5. Recommended Approach: Phased Migration with Data Tiering

Why This Approach for CRM Next + 7 TB?

The combination of a proprietary platform (CRM Next) and massive data volume (7 TB) rules out big-bang. The phased approach with data tiering lets you:

  • Go live with CMS 2.0 quickly (only hot data migrated initially)
  • Keep CRM Next as read-only safety net
  • Migrate archive data in the background without time pressure
  • Validate each tier before moving to the next
  • Negotiate CRM Next license wind-down (reduced from full to read-only)

Data Tiering Strategy

Tier Definition Estimated Size Migration Timing Target System
Hot (Active) Open complaints, active users, current config, last 90 days activity 200-500 GB Phase 1 (pre-cutover) CMS 2.0 Oracle DB
Warm (Recent) Closed complaints (last 2 years), recent communications, recent audit logs 1-2 TB Phase 2 (weeks 1-4 post-cutover) CMS 2.0 DB + OpenSearch
Cold (Archive) Historical complaints (>2 years), old attachments, legacy reports 3-4 TB Phase 3 (months 1-3 post-cutover) Object Storage (S3/MinIO) + Search Index
Frozen (Deep Archive) Very old data, decommissioned modules, system logs 1-2 TB Phase 4 (optional, after CRM Next decommission) Tape/Glacier storage (compliance only)

Migration Waves

Wave 1: Foundation + Hot Data (Weeks 1-8)
  • Set up CMS 2.0 infrastructure (Keycloak, Kafka, API Gateway, OpenSearch)
  • Extract and map CRM Next data model to CMS 2.0 schema
  • Migrate users to Keycloak (typically small, ~1-5 GB)
  • Migrate active/open complaints with their attachments
  • Migrate current business rules to Drools
  • Migrate workflow configurations to Camunda BPMN
  • Go-Live: New complaints start going to CMS 2.0
  • CRM Next set to read-only mode
Wave 2: Warm Data Migration (Weeks 9-16)
  • Background ETL: migrate last 2 years of closed complaints
  • Migrate associated attachments and documents
  • Index historical data in OpenSearch for search
  • Migrate audit trails to CMS 2.0 Audit Service
  • Build unified search across migrated + live data
  • Validate reporting parity with CRM Next reports
Wave 3: Cold Data + Archive (Weeks 17-28)
  • Migrate older historical data to object storage
  • Create search indexes for archive data
  • Build archive access portal in CMS 2.0 (read-only views)
  • Validate all regulatory data is accessible from CMS 2.0
  • Decommission CRM Next read-only instance
  • Terminate CRM Next license
Wave 4: Deep Archive + Cleanup (Weeks 29-36)
  • Move very old data to cold/glacier storage
  • Optimize CMS 2.0 storage (compress, deduplicate)
  • Final data reconciliation and sign-off
  • Destroy CRM Next infrastructure
  • Post-migration performance optimization

6. Phase-wise Migration Plan

Discovery & Data Mapping

4-6 Weeks
  • Engage CRM Next vendor for data export options and support
  • Document CRM Next data model (entities, relationships, custom fields)
  • Map CRM Next entities to CMS 2.0 schema (COMPLAINT_MASTER, etc.)
  • Identify CRM Next workflow rules and map to Camunda/Drools
  • Inventory all integrations consuming CRM Next APIs
  • Profile the 7 TB: what's hot, warm, cold, frozen?
  • Identify data quality issues (orphans, duplicates, corruption)
  • Define data retention policies per tier

Infrastructure & Platform Setup

4-6 Weeks
  • Deploy CMS 2.0 full infrastructure (all 14+ services)
  • Configure Keycloak with CRM Next user role mapping
  • Set up Kafka cluster sized for event volume
  • Provision Oracle/MySQL databases with CMS 2.0 schema
  • Set up object storage for attachments (MinIO/S3)
  • Deploy OpenSearch cluster sized for 7 TB indexing
  • Set up ETL pipeline infrastructure (Apache NiFi/Spark/custom)
  • Configure monitoring and alerting

Business Logic Re-implementation

6-8 Weeks
  • Re-implement CRM Next workflow rules as Camunda BPMN processes
  • Convert CRM Next business rules to Drools DRL files
  • Implement complaint routing logic in Assignment Service
  • Build SLA calculation logic matching CRM Next behavior
  • Implement notification templates matching CRM Next outputs
  • Build reporting queries equivalent to CRM Next reports
  • Shadow-test: compare CMS 2.0 outputs vs CRM Next for same inputs

Hot Data Migration & Cutover

3-4 Weeks
  • Extract active complaints from CRM Next (with vendor support if needed)
  • Transform and load into CMS 2.0 database
  • Copy active attachments to CMS 2.0 storage
  • Migrate users to Keycloak with password reset flow
  • Validate: every active complaint accessible in CMS 2.0
  • CUTOVER: Switch CRM Next to read-only, new work → CMS 2.0
  • 48-hour hypercare with dedicated support team

Warm Data Migration (Background)

4-6 Weeks
  • Run parallel ETL jobs migrating last 2 years of data
  • Batch-copy associated attachments and documents
  • Index migrated records in OpenSearch
  • Build unified search across all data in CMS 2.0
  • Validate: spot-check migrated records, run reconciliation
  • Enable historical data views in CMS 2.0 portal

Cold Data & Archive Migration

6-10 Weeks
  • Migrate 3-4 TB of historical/archive data to object storage
  • Create lightweight search indexes for archive access
  • Build archive viewer in CMS 2.0 portal
  • Validate regulatory compliance: all required data accessible
  • Final reconciliation: compare CRM Next record counts vs CMS 2.0

CRM Next Decommission

2-4 Weeks
  • Confirm zero dependency on CRM Next
  • Take final full backup of CRM Next (for compliance)
  • Terminate CRM Next license
  • Decommission CRM Next servers
  • Update all documentation, runbooks, and architecture diagrams
  • Post-migration cost analysis and performance report

Attachment Migration Strategy (3-4 TB)

Attachments are the largest data category. Strategy options:
Approach Description When to Use
Eager (Full Copy) Copy all attachments to CMS 2.0 storage before cutover When CRM Next is being fully decommissioned immediately
Lazy (On-Access) Copy attachments only when accessed in CMS 2.0. Proxy to CRM Next until copied. When CRM Next remains available and network access is maintained
Tiered (Hot First) Copy active complaint attachments first, background-copy the rest Best balance of speed and completeness
Archive (Cold Storage) Move old attachments to cheap object storage, only hot ones to primary When most attachments are rarely accessed

Data Validation Framework

  • Record Count Validation: Source count == Target count per entity type
  • Checksum Validation: MD5/SHA-256 of critical fields (amounts, dates, IDs)
  • Referential Integrity: All foreign key relationships maintained
  • Sample Audit: Random 1% sample manually verified end-to-end
  • Business Rule Validation: Run same inputs through both systems, compare outputs
  • Attachment Integrity: File size + checksum comparison for migrated files
  • Reconciliation Report: Automated daily report comparing CRM Next vs CMS 2.0

For partners who cannot update immediately, build an adapter layer in the API Gateway:

  • Accept CRM Next API format requests
  • Transform to CMS 2.0 internal format
  • Return responses in CRM Next format
  • Log usage for tracking adapter dependency
  • Set deprecation timeline (3-6 months max)

9. Risk Assessment & Mitigation

Data Loss / Corruption During 7 TB Migration

Impact: Critical | Probability: Medium

Mitigation: Checksum validation per batch, reconciliation reports, CRM Next backup retained for 6 months, incremental approach (not big-bang).

CRM Next Vendor Non-cooperation

Impact: Critical | Probability: Medium

Mitigation: Engage vendor early, contractual data export obligations, prepare for direct DB access as fallback, budget for vendor professional services.

Business Logic Mismatch (CRM Next vs CMS 2.0)

Impact: High | Probability: High

Mitigation: Extended shadow testing period, parallel processing comparison, SME validation of every rule, comprehensive UAT with real scenarios.

In-flight Complaint Disruption

Impact: Critical | Probability: Medium

Mitigation: Migrate only active complaints whose state can be cleanly mapped. Complex in-flight cases stay in CRM Next until resolution.

Extended CRM Next License Cost

Impact: Medium | Probability: High

Mitigation: Negotiate read-only license tier with vendor. Budget for 6-12 month overlap. Calculate break-even point.

Performance Degradation at Scale

Impact: Medium | Probability: Medium

Mitigation: Load test with production-volume data, database indexing strategy, connection pooling, caching, read replicas.

User Adoption Resistance

Impact: Medium | Probability: High

Mitigation: Early user involvement, training program, champion users per team, feedback loops, gradual rollout.

CRM Next Schema Discovery Gaps

Impact: High | Probability: Medium

Mitigation: Extended discovery phase, engage CRM Next consultants, test extraction with sample data before committing to timeline.

Infrastructure Sizing Issues

Impact: Low | Probability: Medium

Mitigation: Cloud-based infrastructure (scale on demand), capacity planning based on CRM Next metrics, auto-scaling policies.

Regulatory Compliance Gap During Transition

Impact: High | Probability: Low

Mitigation: Legal review of data handling during transition, audit trail continuity plan, compliance sign-off per phase.

10. Rollback & Contingency Strategy

Critical Advantage: Because CRM Next remains available (read-only) throughout the migration, rollback is always possible by simply routing traffic back to CRM Next and re-enabling write access.

Rollback Scenarios

Scenario Trigger Rollback Action Data Handling
Pre-cutover failure Data validation fails, business logic mismatch Abort cutover, continue on CRM Next No impact (CRM Next never stopped)
Cutover-day failure Critical bugs in first 48 hours Re-enable CRM Next writes, route traffic back Export complaints created in CMS 2.0 during 48hr window, import to CRM Next
Post-cutover failure (Week 1-4) Data corruption, missing features discovered Route new work back to CRM Next, fix CMS 2.0 Sync CMS 2.0 data back to CRM Next (reverse ETL for affected records)
Post-migration failure (after decommission) Major issue after CRM Next decommissioned Restore CRM Next from backup (longer RTO) Forward-sync all CMS 2.0 data to restored CRM Next

Rollback Prerequisites

  • CRM Next infrastructure remains operational (read-only) for minimum 6 months post-cutover
  • CRM Next full backup taken immediately before cutover and stored securely
  • Reverse ETL scripts prepared and tested (CMS 2.0 → CRM Next format)
  • API Gateway can switch routing in <5 minutes (config change only)
  • Communication template prepared for users if rollback is triggered
  • Maximum data loss acceptable during rollback: defined and agreed (e.g., <1 hour of work)

Point of No Return

The point of no return is when CRM Next is decommissioned (Phase 7).

Before decommission, ensure:

  • All 7 TB data verified accessible in CMS 2.0
  • 30 consecutive days of stable operation
  • Zero critical bugs in backlog
  • All regulatory data accessible from CMS 2.0
  • Stakeholder written sign-off obtained
  • CRM Next final backup archived to long-term storage

Shadow Testing Approach

Critical for business logic validation. Before cutover, run real incoming complaints through both CRM Next and CMS 2.0 in parallel:
  • CRM Next processes normally (primary system)
  • CMS 2.0 receives a copy and processes independently
  • Automated comparator checks: same categorization? Same assignment? Same SLA?
  • Discrepancies logged and investigated
  • Target: <1% discrepancy rate before cutover approval

13. Decision Framework

Key Strategic Decisions

Decision Options Recommendation Rationale
Migrate all 7 TB or only what's needed? Full migration vs tiered vs archive-only for old data Tiered Hot data to primary DB, warm to search index, cold to object storage. Satisfies compliance without overloading primary DB.
When to cut over new complaint intake? After full migration vs after hot data only After hot data Waiting for full 7 TB delays go-live by months. Hot data (active complaints) is sufficient to operate.
How long to keep CRM Next running? Until full migration vs 6 months vs 12 months 6-9 months Need time for warm/cold migration + validation. Negotiate reduced license for read-only access.
In-flight complaints: migrate or complete in CRM Next? Migrate with state mapping vs let them complete in old system Simple states: migrate. Complex: complete in CRM Next Complaints in simple states (NEW, ASSIGNED) can be migrated. Those in complex multi-step workflows should complete where they are.
Attachment migration: eager vs lazy? Copy all upfront vs copy on access vs tiered Tiered Active complaint attachments copied upfront. Old attachments copied in background. Avoids blocking cutover for 3-4 TB file copy.
CRM Next rules: auto-convert or rewrite? Automated conversion vs manual re-implementation Manual re-implementation CRM Next rules are config-based, not code. Auto-conversion is unreliable. Manual gives opportunity to optimize and document.
User migration: same credentials or fresh start? Import passwords vs force password reset Force password reset CRM Next password hashes likely incompatible with Keycloak. Clean start with email-based reset is safer and standard.
Reporting: migrate reports or rebuild? Replicate CRM Next reports vs design new Rebuild from scratch CMS 2.0 has different data model. CRM Next reports won't map cleanly. Opportunity to design better analytics.

Go/No-Go Criteria for Cutover

DO NOT proceed with cutover unless ALL of these are satisfied:
  • Hot data migration complete with zero reconciliation errors
  • Shadow testing shows <1% business logic discrepancy
  • All active complaints accessible and editable in CMS 2.0
  • Load test passed at 2x peak CRM Next traffic
  • Rollback procedure tested successfully in staging
  • All external integration partners confirmed ready
  • User training completed for all affected staff
  • CRM Next set to read-only mode without errors
  • Monitoring and alerting confirmed operational
  • Hypercare team assigned and available for 72 hours
