1. The Problem Explained

Why Individual Services Fail to Build

Each microservice's pom.xml declares this parent:

<parent>
    <groupId>com.rbi.cms</groupId>
    <artifactId>cms-parent</artifactId>
    <version>1.0.0-SNAPSHOT</version>
</parent>

And depends on this shared library:

<dependency>
    <groupId>com.rbi.cms</groupId>
    <artifactId>cms-common</artifactId>
</dependency>

When building locally, Maven finds both via the parent directory's pom.xml (relative path ../pom.xml).

When building on OpenShift, each service is in its own pod/pipeline. There is no parent directory — Maven cannot resolve cms-parent or cms-common, and the build fails with:

ERROR: Non-resolvable parent POM for com.rbi.cms:cms-api-gateway:1.0.0-SNAPSHOT:
  Could not find artifact com.rbi.cms:cms-parent:pom:1.0.0-SNAPSHOT
┌─────────────────────────────────────────────────────────────────────┐ │ LOCAL MACHINE (works) │ │ │ │ CMS2.0/ │ │ ├── pom.xml ← cms-parent (found via filesystem ../pom.xml) │ │ ├── cms-common/ ← built as part of reactor │ │ ├── cms-api-gateway/ ← can reference parent + common │ │ ├── cms-rules-service/ ← can reference parent + common │ │ └── ... │ └─────────────────────────────────────────────────────────────────────┘ ┌─────────────────────────────────────────────────────────────────────┐ │ OPENSHIFT (fails without Nexus) │ │ │ │ Pod: cms-rules-service-build │ │ ├── pom.xml ← references cms-parent... WHERE IS IT? ❌ │ │ └── src/ │ │ │ │ Parent POM is NOT in this pod → Maven fails │ │ cms-common JAR is NOT in this pod → Maven fails │ └─────────────────────────────────────────────────────────────────────┘ ┌─────────────────────────────────────────────────────────────────────┐ │ OPENSHIFT (works WITH Nexus) │ │ │ │ Pod: cms-rules-service-build │ │ ├── pom.xml ← references cms-parent → pulls from Nexus ✅ │ │ ├── settings.xml ← points Maven to Nexus │ │ └── src/ │ │ │ │ Nexus Repository (central to cluster): │ │ ├── com/rbi/cms/cms-parent/1.0.0-SNAPSHOT/cms-parent.pom ✅ │ │ └── com/rbi/cms/cms-common/1.0.0-SNAPSHOT/cms-common.jar ✅ │ └─────────────────────────────────────────────────────────────────────┘

2. Solution Architecture

The Fix: 3-Phase Build Strategy

PhaseWhatWhyWhen
1Deploy Nexus on OpenShiftCentral Maven artifact repository accessible to all build podsOnce (infra setup)
2Publish cms-parent POM to NexusSo child services can resolve their parent POMOn every parent POM change
3Publish cms-common JAR to NexusSo all services can resolve their shared dependencyOn every cms-common change
4Build individual services (parallel)Each service pulls parent + common from NexusOn each service's code change
After phases 1-3 are done (one-time), all 12+ microservices can build independently and in parallel — each in its own pod, each pulling shared dependencies from Nexus.

Create Maven Repositories in Nexus

Via Nexus UI (Settings → Repositories → Create):

Repository NameTypePurpose

4. Step 2: Publish Parent POM to Nexus

This publishes only the POM file (no JAR) — it provides dependency versions, plugin config, and properties to all child services.

From your local machine (or a CI job):

cd CMS2.0

# Publish ONLY the parent POM (not child modules):
mvn deploy -N \
  -DskipTests \
  -DaltDeploymentRepository=nexus-snapshots::default::http://nexus.rbi-cms.svc.cluster.local:8081/repository/maven-snapshots/

# -N = non-recursive (parent only, skip modules)
# This uploads: com/rbi/cms/cms-parent/1.0.0-SNAPSHOT/cms-parent-1.0.0-SNAPSHOT.pom

Using the provided settings.xml:

cd CMS2.0

# Deploy using settings.xml (which already points to Nexus):
mvn deploy -N -DskipTests -s deployment/maven-settings.xml

What gets uploaded to Nexus:

maven-snapshots/
└── com/rbi/cms/
    └── cms-parent/
        └── 1.0.0-SNAPSHOT/
            ├── cms-parent-1.0.0-SNAPSHOT.pom      ← the parent POM
            └── maven-metadata.xml

This is just a POM file (~5KB). No JAR is produced since <packaging>pom</packaging>.

5. Step 3: Publish cms-common to Nexus

This publishes the shared library JAR that all microservices depend on.

cd CMS2.0/cms-common

# Build and publish cms-common:
mvn deploy -DskipTests -s ../deployment/maven-settings.xml

What gets uploaded to Nexus:

maven-snapshots/
└── com/rbi/cms/
    ├── cms-parent/1.0.0-SNAPSHOT/cms-parent.pom      (from Step 2)
    └── cms-common/1.0.0-SNAPSHOT/
        ├── cms-common-1.0.0-SNAPSHOT.jar              ← shared library
        ├── cms-common-1.0.0-SNAPSHOT.pom
        └── maven-metadata.xml

Verify Artifacts in Nexus

# From any pod in the cluster (or your machine if port-forwarded):
curl -s http://nexus.rbi-cms.svc.cluster.local:8081/repository/maven-snapshots/com/rbi/cms/cms-parent/1.0.0-SNAPSHOT/

# Should list the POM file. If empty → publish failed.
After this step, any microservice can be built independently — Maven will pull cms-parent and cms-common from Nexus.

6. Step 4: Microservice Dockerfile (with Nexus)

Each microservice needs a settings.xml file so Maven knows where to find Nexus. The Dockerfile copies it during build.

settings.xml (place in each service's root directory)

<?xml version="1.0" encoding="UTF-8"?>
<settings>
    <mirrors>
        <mirror>
            <id>nexus</id>
            <mirrorOf>*</mirrorOf>
            <url>http://nexus.rbi-cms.svc.cluster.local:8081/repository/maven-public/</url>
        </mirror>
    </mirrors>
    <servers>
        <server>
            <id>nexus</id>
            <username>admin</username>
            <password>nexus_admin_password</password>
        </server>
    </servers>
    <profiles>
        <profile>
            <id>nexus</id>
            <repositories>
                <repository>
                    <id>nexus</id>
                    <url>http://nexus.rbi-cms.svc.cluster.local:8081/repository/maven-public/</url>
                    <releases><enabled>true</enabled></releases>
                    <snapshots><enabled>true</enabled></snapshots>
                </repository>
            </repositories>
        </profile>
    </profiles>
    <activeProfiles>
        <activeProfile>nexus</activeProfile>
    </activeProfiles>
</settings>

Dockerfile (for each microservice)

FROM eclipse-temurin:21-jdk-alpine AS build
WORKDIR /app

# Copy Maven settings pointing to Nexus (so parent + common can be resolved)
COPY settings.xml /root/.m2/settings.xml

# Copy only pom.xml first (Docker layer caching for deps)
COPY pom.xml .
RUN apk add --no-cache maven && mvn dependency:go-offline -q || true

# Copy source and build
COPY src ./src
RUN mvn clean package -DskipTests -q

# ─── Runtime Stage ───
FROM eclipse-temurin:21-jre-alpine
WORKDIR /app
RUN addgroup -S cms && adduser -S cms -G cms

COPY --from=build /app/target/*.jar app.jar
USER cms

EXPOSE 8080

HEALTHCHECK --interval=30s --timeout=5s --start-period=60s --retries=3 \
  CMD wget -qO- http://localhost:8080/actuator/health || exit 1

ENTRYPOINT ["java", \
  "-XX:+UseG1GC", \
  "-XX:MaxGCPauseMillis=200", \
  "-XX:InitialRAMPercentage=50.0", \
  "-XX:MaxRAMPercentage=75.0", \
  "-Djava.security.egd=file:/dev/./urandom", \
  "-jar", "app.jar"]

Key Line Explained

COPY settings.xml /root/.m2/settings.xml

This is what makes it work. When Maven runs mvn package, it reads /root/.m2/settings.xml, discovers Nexus as a mirror, and downloads cms-parent POM + cms-common JAR from there — just like it would download Spring Boot starters from Maven Central.

What Each Service Directory Should Contain:

cms-api-gateway/
├── Dockerfile          ← uses settings.xml to resolve parent from Nexus
├── settings.xml        ← points to Nexus (copy from deployment/maven-settings.xml)
├── pom.xml             ← references cms-parent as parent
└── src/
    └── main/java/...

Quick Copy Script (copy settings.xml to all services)

cd CMS2.0

SERVICES="cms-api-gateway cms-assignment-service cms-audit-service \
  cms-eligibility-service cms-ingestion-service cms-notification-service \
  cms-outbox-publisher cms-rules-service cms-search-service \
  cms-sla-monitor-service cms-storage-service cms-workflow-service"

for svc in $SERVICES; do
  cp deployment/maven-settings.xml $svc/settings.xml
  echo "✓ Copied settings.xml to $svc/"
done

7. Step 5: Build & Deploy Order

Critical Build Order

Phase 1: Infrastructure (Nexus, Oracle, Kafka, Keycloak, OpenSearch) │ ▼ Phase 2: Publish to Nexus (MUST complete before Phase 3) ├── mvn deploy -N (cms-parent POM) └── mvn deploy (cms-common JAR) │ ▼ Phase 3: Build ALL microservices (can run in PARALLEL) ├── cms-api-gateway ──┐ ├── cms-assignment-service ──┤ ├── cms-audit-service ──┤ ├── cms-eligibility-service ──┤ ├── cms-ingestion-service ──┤ All pull parent + common ├── cms-notification-service──┤ from Nexus automatically ├── cms-outbox-publisher ──┤ ├── cms-rules-service ──┤ ├── cms-search-service ──┤ ├── cms-sla-monitor-service ──┤ ├── cms-storage-service ──┤ └── cms-workflow-service ──┘ │ ▼ Phase 4: Build standalone services (no Nexus dependency) ├── cms-backend (has its own Spring Boot parent) ├── cms-portal-frontend (Angular, no Maven) ├── cms-frontend (Angular, no Maven) └── cms-paddle-ocr (Python, no Maven) │ ▼ Phase 5: Deploy all to OpenShift
cms-backend is standalone — its pom.xml uses spring-boot-starter-parent directly, NOT cms-parent. It can be built without Nexus.

Build Images Command Sequence

# === PHASE 2: Publish shared artifacts to Nexus ===
cd CMS2.0

# Publish parent POM:
mvn deploy -N -DskipTests -s deployment/maven-settings.xml
echo "✓ cms-parent published to Nexus"

# Publish cms-common:
cd cms-common
mvn deploy -DskipTests -s ../deployment/maven-settings.xml
echo "✓ cms-common published to Nexus"
cd ..

# === PHASE 3: Build microservice images (parallel) ===
REGISTRY="image-registry.openshift-image-registry.svc:5000/rbi-cms"

# These can all run in parallel:
podman build -t $REGISTRY/cms-api-gateway:latest         cms-api-gateway/ &
podman build -t $REGISTRY/cms-assignment-service:latest   cms-assignment-service/ &
podman build -t $REGISTRY/cms-audit-service:latest        cms-audit-service/ &
podman build -t $REGISTRY/cms-eligibility-service:latest  cms-eligibility-service/ &
podman build -t $REGISTRY/cms-ingestion-service:latest    cms-ingestion-service/ &
podman build -t $REGISTRY/cms-notification-service:latest cms-notification-service/ &
podman build -t $REGISTRY/cms-outbox-publisher:latest     cms-outbox-publisher/ &
podman build -t $REGISTRY/cms-rules-service:latest        cms-rules-service/ &
podman build -t $REGISTRY/cms-search-service:latest       cms-search-service/ &
podman build -t $REGISTRY/cms-sla-monitor-service:latest  cms-sla-monitor-service/ &
podman build -t $REGISTRY/cms-storage-service:latest      cms-storage-service/ &
podman build -t $REGISTRY/cms-workflow-service:latest      cms-workflow-service/ &
wait
echo "✓ All microservice images built"

# === PHASE 4: Build standalone services ===
podman build -t $REGISTRY/cms-backend:latest              cms-backend/ &
podman build -t $REGISTRY/cms-portal-frontend:latest      cms-portal-frontend/ &
podman build -t $REGISTRY/cms-paddle-ocr:latest           cms-paddle-ocr/ &
wait
echo "✓ Standalone services built"

# === PUSH ALL IMAGES ===
for img in cms-api-gateway cms-assignment-service cms-audit-service \
  cms-eligibility-service cms-ingestion-service cms-notification-service \
  cms-outbox-publisher cms-rules-service cms-search-service \
  cms-sla-monitor-service cms-storage-service cms-workflow-service \
  cms-backend cms-portal-frontend cms-paddle-ocr; do
    podman push $REGISTRY/$img:latest
    echo "  ✓ Pushed $img"
done

8. Step 6: Full Deployment Script

deploy-services.sh (run after images are pushed)

#!/bin/bash
set -e
OPENSHIFT_DIR="$(dirname "$0")/openshift"

echo "============================================="
echo "  CMS 2.0 - OpenShift Service Deployment"
echo "============================================="

# Verify logged in
oc whoami || { echo "ERROR: Not logged in. Run 'oc login' first."; exit 1; }

echo ""
echo "[Phase 1] Namespace & ConfigMaps & Secrets"
echo "---------------------------------------------"
oc apply -f $OPENSHIFT_DIR/namespace.yaml
oc apply -f $OPENSHIFT_DIR/configmap-common.yaml
oc apply -f $OPENSHIFT_DIR/secret-db.yaml
oc apply -f $OPENSHIFT_DIR/secret-keycloak.yaml
oc apply -f $OPENSHIFT_DIR/secret-sms.yaml
oc apply -f $OPENSHIFT_DIR/secret-opensearch.yaml
oc apply -f $OPENSHIFT_DIR/secret-kafka.yaml
oc apply -f $OPENSHIFT_DIR/pvc.yaml
echo "  ✓ Foundation applied"

echo ""
echo "[Phase 2] Core Services (no inter-service deps)"
echo "---------------------------------------------"
oc apply -f $OPENSHIFT_DIR/deployment-audit-service.yaml
oc apply -f $OPENSHIFT_DIR/deployment-storage-service.yaml
oc apply -f $OPENSHIFT_DIR/deployment-paddle-ocr.yaml
oc apply -f $OPENSHIFT_DIR/deployment-notification-service.yaml
oc apply -f $OPENSHIFT_DIR/deployment-search-service.yaml
echo "  ✓ Core services deployed"

echo ""
echo "[Phase 3] Business Logic Services"
echo "---------------------------------------------"
oc apply -f $OPENSHIFT_DIR/deployment-rules-service.yaml
oc apply -f $OPENSHIFT_DIR/deployment-eligibility-service.yaml
oc apply -f $OPENSHIFT_DIR/deployment-sla-monitor-service.yaml
oc apply -f $OPENSHIFT_DIR/deployment-assignment-service.yaml
oc apply -f $OPENSHIFT_DIR/deployment-workflow-service.yaml
oc apply -f $OPENSHIFT_DIR/deployment-ingestion-service.yaml
oc apply -f $OPENSHIFT_DIR/deployment-outbox-publisher.yaml
echo "  ✓ Business services deployed"

echo ""
echo "[Phase 4] Backend & API Gateway"
echo "---------------------------------------------"
oc apply -f $OPENSHIFT_DIR/deployment-backend.yaml
oc apply -f $OPENSHIFT_DIR/deployment-api-gateway.yaml
echo "  ✓ Backend + Gateway deployed"

echo ""
echo "[Phase 5] Frontends"
echo "---------------------------------------------"
oc apply -f $OPENSHIFT_DIR/deployment-portal-frontend.yaml
oc apply -f $OPENSHIFT_DIR/deployment-frontend.yaml
echo "  ✓ Frontends deployed"

echo ""
echo "============================================="
echo "  All 16 services deployed!"
echo "  Run: oc get pods -n rbi-cms"
echo "============================================="

9. Step 7: Tekton CI/CD Pipeline (Automated)

For fully automated builds on every git push, use this updated Tekton pipeline that includes Nexus resolution.

Pre-requisite: Maven Settings as ConfigMap

# Create ConfigMap with the settings.xml:
oc create configmap maven-settings \
  --from-file=settings.xml=deployment/maven-settings.xml \
  -n rbi-cms

Updated Pipeline: cms-java-service-pipeline.yaml

apiVersion: tekton.dev/v1beta1
kind: Pipeline
metadata:
  name: cms-java-service-pipeline
  namespace: rbi-cms
spec:
  params:
    - name: SERVICE_NAME
      type: string
    - name: GIT_URL
      type: string
    - name: GIT_REVISION
      type: string
      default: main
    - name: IMAGE_TAG
      type: string
      default: latest
  workspaces:
    - name: shared-workspace
    - name: maven-settings

  tasks:
    - name: fetch-source
      taskRef:
        name: git-clone
        kind: ClusterTask
      workspaces:
        - name: output
          workspace: shared-workspace
      params:
        - name: url
          value: $(params.GIT_URL)
        - name: revision
          value: $(params.GIT_REVISION)

    - name: copy-maven-settings
      taskRef:
        name: openshift-client
        kind: ClusterTask
      runAfter: [fetch-source]
      params:
        - name: SCRIPT
          value: |
            # Copy settings.xml into the workspace for Maven to use
            cp /workspace/maven-settings/settings.xml /workspace/source/settings.xml
            # Also place in .m2 for the build stage
            mkdir -p /workspace/source/.m2
            cp /workspace/maven-settings/settings.xml /workspace/source/.m2/settings.xml

    - name: build-and-test
      taskRef:
        name: maven
        kind: ClusterTask
      runAfter: [copy-maven-settings]
      workspaces:
        - name: source
          workspace: shared-workspace
        - name: maven-settings
          workspace: maven-settings
      params:
        - name: GOALS
          value:
            - clean
            - package
            - -DskipTests=false
        - name: MAVEN_IMAGE
          value: maven:3.9-eclipse-temurin-21-alpine

    - name: build-push-image
      taskRef:
        name: buildah
        kind: ClusterTask
      runAfter: [build-and-test]
      workspaces:
        - name: source
          workspace: shared-workspace
      params:
        - name: IMAGE
          value: image-registry.openshift-image-registry.svc:5000/rbi-cms/$(params.SERVICE_NAME):$(params.IMAGE_TAG)
        - name: DOCKERFILE
          value: ./Dockerfile

    - name: deploy-rollout
      taskRef:
        name: openshift-client
        kind: ClusterTask
      runAfter: [build-push-image]
      params:
        - name: SCRIPT
          value: |
            oc rollout restart deployment/$(params.SERVICE_NAME) -n rbi-cms
            oc rollout status deployment/$(params.SERVICE_NAME) -n rbi-cms --timeout=300s

Trigger a Pipeline Run

# Example: Build and deploy cms-rules-service
cat <<EOF | oc apply -f -
apiVersion: tekton.dev/v1beta1
kind: PipelineRun
metadata:
  generateName: cms-rules-service-run-
  namespace: rbi-cms
spec:
  pipelineRef:
    name: cms-java-service-pipeline
  params:
    - name: SERVICE_NAME
      value: cms-rules-service
    - name: GIT_URL
      value: https://github.com/your-org/cms-rules-service.git
    - name: GIT_REVISION
      value: main
    - name: IMAGE_TAG
      value: "1.0.0"
  workspaces:
    - name: shared-workspace
      volumeClaimTemplate:
        spec:
          accessModes: [ReadWriteOnce]
          resources:
            requests:
              storage: 2Gi
    - name: maven-settings
      configMap:
        name: maven-settings
EOF

Troubleshooting: "Could not find artifact cms-parent"

  1. Verify Nexus is accessible from build pod: 
    oc exec deployment/nexus -n rbi-cms -- curl -s http://localhost:8081/service/rest/v1/status
  2. Verify parent is published: 
    curl http://nexus-route/repository/maven-snapshots/com/rbi/cms/cms-parent/1.0.0-SNAPSHOT/
  3. Verify settings.xml is correct in the service directory: 
    cat cms-api-gateway/settings.xml | grep -A2 "mirror"
# Should show: http://nexus.rbi-cms.svc.cluster.local:8081/repository/maven-public/
  4. Verify Dockerfile copies settings.xml: 
    grep "settings.xml" cms-api-gateway/Dockerfile
# Should show: COPY settings.xml /root/.m2/settings.xml
  5. Re-publish if needed: 
    mvn deploy -N -DskipTests -s deployment/maven-settings.xml

Troubleshooting: "Could not find artifact cms-common"

  1. Ensure cms-common was published AFTER the parent POM (it depends on the parent).
  2. Re-publish: cd cms-common && mvn deploy -DskipTests -s ../deployment/maven-settings.xml
  3. Check Nexus UI → Browse → maven-snapshots → com/rbi/cms/cms-common/ → should have .jar file

Troubleshooting: Nexus unreachable from build pod

# Check Nexus service exists:
oc get svc nexus -n rbi-cms

# Check DNS resolution from a build pod:
oc run test-dns --rm -it --image=alpine --restart=Never -- \
  nslookup nexus.rbi-cms.svc.cluster.local

# If using external Nexus, update settings.xml URL accordingly
