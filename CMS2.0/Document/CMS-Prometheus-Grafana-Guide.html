1 What is Prometheus & Grafana?

▼

Prometheus

  • Open-source metrics collection & monitoring system (originally built at SoundCloud, now a CNCF graduated project)
  • Pulls (scrapes) metrics from services at regular intervals (pull-based model)
  • Stores time-series data: metric_name{label1="value1"} value timestamp
  • Has its own powerful query language: PromQL
  • Role: "Data Collector" — knows all the numbers but does not visualize them well

Grafana

  • Open-source visualization & dashboarding platform
  • Connects to multiple data sources (Prometheus, MySQL, Elasticsearch, InfluxDB, etc.)
  • Creates beautiful charts, gauges, tables, heatmaps, and alerts
  • Role: "Visualizer" — makes Prometheus data human-readable and actionable

How They Work Together

Your CMS App
Exposes /actuator/prometheus
➡
Prometheus
Scrapes every 15s, stores time-series
➡
Grafana
Queries Prometheus, renders dashboards
Flow # The complete data flow: Your App → exposes /actuator/prometheus endpoint (raw metrics in text format) Prometheus → scrapes these endpoints every 15s → stores as time-series data Grafana → queries Prometheus via PromQL → renders beautiful dashboards
Real-World Analogy

Prometheus = A thermometer that records the temperature every 15 seconds and writes it in a notebook.

Grafana = The TV screen in the control room that shows you the temperature graph over time, with color-coded alerts when it gets too hot.

Together = You get continuous monitoring with instant visibility. Without Prometheus, there is no data. Without Grafana, the data stays hidden in a notebook nobody reads.

Spring Boot Actuator Configuration

Every CMS service already has spring-boot-starter-actuator + micrometer-registry-prometheus in pom.xml. This exposes a Prometheus-compatible metrics endpoint automatically.

Start Prometheus

Port: 9090

4

Grafana Data Source Configuration

Grafana needs to know WHERE to get data from. We added Prometheus as a data source via API.

JSON { "name": "Prometheus", "type": "prometheus", "url": "http://localhost:9090", "access": "proxy", // Grafana backend calls Prometheus (not browser) "isDefault": true }
What does "access: proxy" mean?

It means the Grafana server (backend) makes HTTP requests to Prometheus on behalf of your browser. This is more secure than "direct" mode where the browser calls Prometheus directly (which would require network access from client machines).

6 What Else You Can Do

▼

6.1 Add Alerts in Grafana

Grafana can fire alerts when metrics cross thresholds, notifying your team immediately.

  • Set threshold: "Alert if response time > 2s for 5 minutes"
  • Set notification: Email, Slack, Microsoft Teams, or Webhook when alert fires
  • How to create: Panel → Edit → Alert tab → Create alert rule
Example Alert Conditions
  • CPU usage > 80% for 5 minutes
  • Heap memory > 90% of max
  • Error rate (5xx) > 5 per minute
  • Service DOWN for more than 30 seconds
  • P95 response time > 3 seconds

6.2 Add Custom Metrics in Code

You can instrument your business logic with custom counters, gauges, and timers:

Java import io.micrometer.core.instrument.MeterRegistry; import io.micrometer.core.instrument.Counter; import io.micrometer.core.instrument.Timer; @Service public class ComplaintService { private final MeterRegistry meterRegistry; public ComplaintService(MeterRegistry meterRegistry) { this.meterRegistry = meterRegistry; } public void registerComplaint(String category) { // Business logic... // Increment a counter with labels meterRegistry.counter("complaints.registered", "category", category, "source", "portal" ).increment(); } public void resolveComplaint(String complaintId) { Timer.Sample sample = Timer.start(meterRegistry); // ... resolution logic ... sample.stop(meterRegistry.timer("complaints.resolution.time", "type", "manual")); } }
How Custom Metrics Propagate

Once you add the code above, the metric automatically appears at /actuator/prometheus. Prometheus auto-scrapes it. Then you just write a PromQL query in Grafana. Zero additional configuration needed.

6.3 Add More Dashboards

  • JVM Deep Dive: GC details, class loading, buffer pools, thread states
  • Kafka Consumer Dashboard: Consumer lag, throughput, error rates, rebalancing events
  • Business Metrics: Complaints per hour, resolution rate, SLA compliance percentage
  • Import community dashboards: Grafana Dashboard ID 4701 (JVM Micrometer) — one-click import!
Importing a Community Dashboard

In Grafana: Dashboards → Import → Enter ID 4701 → Select Prometheus data source → Import. Instantly get a production-ready JVM monitoring dashboard.

6.4 Log Aggregation (Future Enhancement)

  • Add Loki for centralized log collection (Prometheus for logs)
  • Grafana can query logs alongside metrics in the same dashboard
  • Correlate: See a "spike in errors" on a graph → click → jump directly to the actual error logs from that time window
  • Stack: Promtail (agent) → Loki (storage) → Grafana (query & display)

7 Configuration Update Steps

▼

When You Add a New Microservice

1

Add actuator dependency to pom.xml

XML <dependency> <groupId>org.springframework.boot</groupId> <artifactId>spring-boot-starter-actuator</artifactId> </dependency> <dependency> <groupId>io.micrometer</groupId> <artifactId>micrometer-registry-prometheus</artifactId> </dependency>
2

Expose prometheus endpoint in application.yml

YAML management: endpoints: web: exposure: include: health,info,metrics,prometheus
3

Add new job to prometheus.yml

YAML - job_name: "cms-new-service" metrics_path: /cms-new-service/actuator/prometheus static_configs: - targets: ["localhost:8088"]
4

Restart Prometheus

Stop the running Prometheus process (Ctrl+C) and start again with the same command. Alternatively, send a SIGHUP or POST to /-/reload if --web.enable-lifecycle flag was set.

5

(Optional) Add panels in Grafana dashboard

The new service's metrics will auto-appear in existing panels that use job=~"cms-.*" regex. For service-specific panels, add new ones via the Grafana UI.

When You Change a Service Port

  1. Update server.port in the service's application.yml
  2. Update targets in prometheus.yml to match the new port
  3. Restart Prometheus (or hot-reload if lifecycle API is enabled)

When You Want Custom Metrics

  1. Add Micrometer counter/gauge/timer in your Java code (see Section 6.2)
  2. Metrics automatically appear at /actuator/prometheus (no restart needed for new metrics)
  3. Prometheus auto-scrapes them on next interval
  4. Add a PromQL query in a new Grafana panel to visualize

When You Want Alerts

  1. In Grafana: Dashboard → Panel → Edit → Alert tab
  2. Set condition (e.g., query(A, 5m, now) IS ABOVE 2)
  3. Set evaluation interval (how often to check: e.g., every 1 minute)
  4. Set notification channel (email, Slack webhook, Microsoft Teams, PagerDuty)
  5. Save the dashboard — alerts are now active

8 PromQL Quick Reference

▼

PromQL (Prometheus Query Language) is how you ask Prometheus for specific data. Here are the most useful queries for CMS monitoring:

QueryWhat It Shows
up{job=~"cms-.*"} Which CMS services are running (1 = UP, 0 = DOWN)
rate(http_server_requests_seconds_count[5m]) Requests per second (averaged over last 5 minutes)
histogram_quantile(0.95, rate(http_server_requests_seconds_bucket[5m])) 95th percentile response time (95% of requests are faster than this)
jvm_memory_used_bytes{area="heap"} / 1024 / 1024 Current heap memory usage in megabytes
increase(http_server_requests_seconds_count{status="500"}[1h]) Total number of 500 Internal Server Errors in the last hour
process_uptime_seconds / 3600 Service uptime in hours
rate(jvm_gc_pause_seconds_sum[5m]) Rate of time spent in garbage collection
sum(rate(http_server_requests_seconds_count[5m])) by (job) Total request rate grouped by service
topk(3, sum(rate(http_server_requests_seconds_count[5m])) by (uri)) Top 3 busiest API endpoints
http_server_requests_seconds_sum / http_server_requests_seconds_count Average response time per endpoint

PromQL Concepts

FunctionPurposeExample
rate()Per-second average rate of increase (for counters)rate(requests_total[5m])
increase()Total increase over time rangeincrease(errors_total[1h])
histogram_quantile()Calculate percentiles from histogramhistogram_quantile(0.99, ...)
sum() by (label)Aggregate and group resultssum(up) by (job)
topk(n, ...)Top N time series by valuetopk(5, rate(...))
avg_over_time()Average value over a time windowavg_over_time(cpu[1h])
[5m]Range vector selector (last 5 minutes of data)Used inside rate(), increase()
Think of PromQL Like This

rate() = "How fast is it happening?" (speed)

increase() = "How much happened total?" (count over time)

histogram_quantile() = "How slow is the slowest X%?" (percentile)

sum() by (job) = "Group by service and add them up" (aggregation)

9 Troubleshooting

▼
ProblemCauseFix
Target shows "DOWN" in Prometheus Service not running, wrong port, or incorrect metrics_path 1. Check if service is running: netstat -ano | findstr :8082
2. Verify metrics_path matches your context-path
3. Test URL manually in browser: http://localhost:8082/cms-ingestion/actuator/prometheus
No data in Grafana panel Wrong datasource selected, time range too narrow, or query syntax error 1. Check datasource is set to "Prometheus" (not default)
2. Expand time range to "Last 1 hour" or "Last 6 hours"
3. Test query directly in Prometheus UI (localhost:9090/graph)
Metrics not appearing at /actuator/prometheus Actuator endpoint not exposed or micrometer dependency missing 1. Verify management.endpoints.web.exposure.include contains prometheus
2. Verify micrometer-registry-prometheus is in pom.xml
3. Run mvn dependency:tree | grep micrometer
Prometheus won't start YAML syntax error in prometheus.yml Validate config: promtool check config prometheus.yml
Common issues: tabs instead of spaces, missing colon, incorrect indentation
Grafana login fails Forgot password or DB corruption Reset by deleting: C:\tools\grafana-v11.3.0\data\grafana.db
Restart Grafana. Default credentials: admin / admin
Prometheus port 9090 in use Another instance running or port conflict Kill existing: taskkill /F /PID <pid>
Or change port: ./prometheus.exe --web.listen-address=:9091
Grafana shows "No data" for new service Prometheus hasn't scraped yet or job_name filter doesn't match 1. Wait 15-30 seconds for first scrape
2. Check Prometheus targets page shows UP
3. Verify PromQL uses correct job label: up{job="your-job-name"}
Common Gotcha: Context Path Mismatch

The most common issue is mismatch between the metrics_path in prometheus.yml and the actual context-path of your service. For example, if your service has server.servlet.context-path=/cms-ingestion, then the metrics_path must be /cms-ingestion/actuator/prometheus, NOT just /actuator/prometheus.
