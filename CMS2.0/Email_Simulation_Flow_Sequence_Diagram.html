2. Step-by-Step Technical Details

# Step Component / Service Action
1 Compose email EmailSimulationComponent User fills fromEmail, fromName, subject, body. Input sanitized (XSS check, email format validation)
2 Send email CmsService.simulateIncomingEmail() POST to /api/email-simulation/receive with sanitized payload
3 Save inbound EmailSimulationService.receiveEmail() Creates SimulatedEmail (INBOUND, PROCESSED) with new threadId + complaintNumber
4 Create complaint ComplaintRepository.save() Complaint created: status=awaiting_details, priority=medium, filingType=email
5 Auto-reply EmailSimulationService Creates OUTBOUND SimulatedEmail with reply body + attachmentUrl (form template link)
6 Timeline ComplaintTimelineRepository Records: action=filed, performedBy="Email System", toStatus=awaiting_details
7-8 Show thread Component view='thread' Displays email thread UI with inbound + auto-reply messages
9-11 Download form downloadComplaintForm() Generates HTML form as Blob (pre-filled with sender info, banks, categories, states). Uses file-saver to trigger download. No server call.
12 Fill form User (offline) Opens downloaded .html in browser, fills fields, clicks embedded "Save Filled Form" JS button
13-14 Upload & parse parseHtmlForm(file) Reads file as text → DOMParser → extracts values by element ID (complainantName, phone, etc.). No server call.
15-16 Submit reply CmsService.replyWithForm() POST to /api/email-simulation/reply-with-form with extracted form data + threadId
17 Complete complaint EmailSimulationService.receiveFormReply() Updates Complaint with full details (name, phone, address, bank, category, description). Status → pending. Saves form-reply + confirmation emails to DB.
18-19 Show result Component Full thread displayed with 4 emails (inbound → auto-reply → form-reply → confirmation)

4. API Endpoints (EmailSimulationController)

Method Endpoint Purpose Returns
POST /api/email-simulation/receive Simulate incoming email → creates complaint + auto-reply {threadId, complaintNumber, status, emails[]}
POST /api/email-simulation/reply-with-form Submit filled form data → completes complaint {threadId, complaintNumber, status: COMPLETED, emails[]}
GET /api/email-simulation/threads List all email threads with status [{threadId, complaintNumber, fromEmail, subject, emailCount, status}]
GET /api/email-simulation/threads/{threadId} Get single thread with all emails {threadId, complaintNumber, status, emails[]}
GET /api/email-simulation/stats Dashboard stats (cached in Redis) {totalThreads, awaitingForm, completed}
GET /api/email-simulation/form-template/{complaintNumber} Form field definitions (metadata only) {complaintNumber, complainantEmail, fields[]}
GET /api/email-simulation/inbox All inbound emails SimulatedEmail[]
GET /api/email-simulation/sent All outbound emails SimulatedEmail[]

6. Design Decisions

Why Simulation (No SMTP)?

  • Development speed — No mail server setup required; works fully offline
  • Testability — Entire email flow testable without external dependencies
  • Demo-friendly — Can demonstrate end-to-end complaint-via-email flow in any environment
  • Form parsing client-side — Avoids server dependency for HTML parsing; user sees preview before submit
  • Thread status derived from email count — Simple logic: 1 inbound = AWAITING, 2+ = COMPLETED
  • Redis caching on stats — @Cacheable("email-stats") with @CacheEvict on writes
  • Security — Input sanitization (XSS, email format, length) applied before sending to backend
