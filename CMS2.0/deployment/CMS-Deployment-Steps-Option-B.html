1

Nexus Repository Setup

Set up Sonatype Nexus (or any Maven-compatible repository manager) to host the shared artifacts.

1.1 Create Repositories in Nexus

Repository NameTypePurpose
cms-releasesmaven2 (hosted, release)Stable release artifacts
cms-snapshotsmaven2 (hosted, snapshot)Development snapshots
cms-groupmaven2 (group)Aggregates releases + snapshots + maven-central proxy

1.2 Create a Deploy User

Create a Nexus user (e.g., cms-deployer) with privileges to deploy to cms-releases and cms-snapshots.

1.3 Maven settings.xml

This file is already provided at deployment/maven-settings.xml. Copy it to each microservice repo:

<settings>
  <servers>
    <server>
      <id>nexus-releases</id>
      <username>${env.NEXUS_USER}</username>
      <password>${env.NEXUS_PASSWORD}</password>
    </server>
    <server>
      <id>nexus-snapshots</id>
      <username>${env.NEXUS_USER}</username>
      <password>${env.NEXUS_PASSWORD}</password>
    </server>
  </servers>
  <mirrors>
    <mirror>
      <id>nexus</id>
      <mirrorOf>*</mirrorOf>
      <url>https://your-nexus.example.com/repository/cms-group/</url>
    </mirror>
  </mirrors>
  <profiles>
    <profile>
      <id>nexus</id>
      <repositories>
        <repository>
          <id>central</id>
          <url>https://your-nexus.example.com/repository/cms-group/</url>
        </repository>
      </repositories>
    </profile>
  </profiles>
  <activeProfiles>
    <activeProfile>nexus</activeProfile>
  </activeProfiles>
</settings>
⚠️
Important: Replace https://your-nexus.example.com with your actual Nexus URL. Set NEXUS_USER and NEXUS_PASSWORD as environment variables in your CI/CD pipeline.
2

Publish cms-parent to Nexus

The root pom.xml (cms-parent) defines dependency versions and plugin management for all microservices. It must be published to Nexus first.

2.1 Create the cms-parent Repo

# Create a new repo with just the parent pom
mkdir cms-parent && cd cms-parent
git init

# Copy the root pom.xml from the monorepo
cp /path/to/CMS2.0/pom.xml ./pom.xml

git add pom.xml
git commit -m "feat: initial cms-parent POM"
git remote add origin https://your-git-server/cms-parent.git
git push -u origin main

2.2 Deploy to Nexus

# From cms-parent repo root:
export NEXUS_USER=cms-deployer
export NEXUS_PASSWORD=<your-password>

mvn deploy -N -s settings.xml

# -N = non-recursive (deploys only the parent POM, no submodules)

2.3 Verify

# Check the artifact is available:
curl -u $NEXUS_USER:$NEXUS_PASSWORD \
  "https://your-nexus.example.com/repository/cms-releases/com/hrms/cms/cms-parent/1.0.0/cms-parent-1.0.0.pom"

# Should return HTTP 200 with POM content
✅
Success Criteria: cms-parent:1.0.0 is available in Nexus. All microservice POMs reference this as their <parent>.

Publish cms-common to Nexus

The cms-common library contains shared DTOs, enums, exceptions, and utility classes used by all 12 microservices.

3.1 Create the cms-common Repo

# Create a new repo from the cms-common module
mkdir cms-common && cd cms-common
git init

# Copy everything from the monorepo's cms-common directory
cp -r /path/to/CMS2.0/cms-common/* .
cp /path/to/CMS2.0/deployment/maven-settings.xml ./settings.xml

git add .
git commit -m "feat: initial cms-common library"
git remote add origin https://your-git-server/cms-common.git
git push -u origin main

3.2 Deploy to Nexus

export NEXUS_USER=cms-deployer
export NEXUS_PASSWORD=<your-password>

mvn deploy -s settings.xml

# This builds the JAR and publishes:
#   com.hrms.cms:cms-common:1.0.0 → nexus-releases

3.3 Verify

curl -u $NEXUS_USER:$NEXUS_PASSWORD \
  "https://your-nexus.example.com/repository/cms-releases/com/hrms/cms/cms-common/1.0.0/cms-common-1.0.0.jar"

# Should return HTTP 200 (or download the JAR)
ℹ️
Note: Any change to cms-common requires a version bump and re-deploy to Nexus, then update the version in consuming microservices.

4.1 Steps for Each Service

# Repeat for each service (example: cms-rules-service)
SERVICE=cms-rules-service

mkdir $SERVICE && cd $SERVICE
git init

# Copy source from monorepo
cp -r /path/to/CMS2.0/$SERVICE/* .
cp /path/to/CMS2.0/$SERVICE/.dockerignore .

# Copy settings.xml for Nexus access during Docker build
cp /path/to/CMS2.0/deployment/maven-settings.xml ./settings.xml

git add .
git commit -m "feat: initial $SERVICE from monorepo"
git remote add origin https://your-git-server/$SERVICE.git
git push -u origin main

4.2 Expected Repo Structure

cms-rules-service/
├── pom.xml              ← parent = cms-parent (from Nexus)
├── src/
│   └── main/java/...
├── Dockerfile           ← Multi-stage build with settings.xml
├── .dockerignore
└── settings.xml         ← Nexus credentials for maven build

4.3 Dockerfile (Already Configured)

Each microservice Dockerfile already includes:

# Inside Dockerfile (build stage):
COPY settings.xml /root/.m2/settings.xml
RUN mvn package -DskipTests -s /root/.m2/settings.xml

# This pulls cms-parent + cms-common from Nexus during build
🚨
Critical: Nexus must be accessible from the build environment (CI runner or OpenShift build pod). If behind a firewall, configure network policies accordingly.

6.5 How Frontend API Routing Works

ℹ️

In OpenShift, environment.openshift.ts sets apiBaseUrl: '' (empty string). Angular makes requests to relative path /api/.... Nginx receives these requests and proxies them to http://${API_BACKEND_HOST}:${API_BACKEND_PORT}/api/....

The docker-entrypoint.sh script uses envsubst to substitute the env vars into the nginx config at container startup.

DevOps Creates Pipelines

DevOps creates a Tekton (or Jenkins/GitHub Actions) pipeline for each service. Templates are provided in deployment/tekton/.

7.1 Pipeline Per Service Type

Service TypePipeline TemplateSteps
Java Microservice (×12) pipeline-java-service.yaml git-clone → maven-test → maven-package → buildah-push → oc-rollout
Angular Frontend (×2) pipeline-angular-frontend.yaml git-clone → npm-install → ng-build → buildah-push → oc-rollout
Python (×1) pipeline-python-ocr.yaml git-clone → buildah-push → oc-rollout
cms-backend (×1) pipeline-java-service.yaml Same as microservice (no Nexus dependency)

7.2 Pipeline Configuration

Each pipeline needs these parameters:

# PipelineRun parameters:
params:
  - name: GIT_REPO_URL
    value: "https://your-git-server/cms-rules-service.git"
  - name: GIT_REVISION
    value: "main"
  - name: IMAGE_NAME
    value: "image-registry.openshift-image-registry.svc:5000/rbi-cms/cms-rules-service"
  - name: IMAGE_TAG
    value: "$(tt.params.git-revision)"  # or "latest"

7.3 Nexus Credentials in Pipeline

For microservices (not cms-backend or frontends), the pipeline must provide Nexus credentials during the Maven build step:

# In Tekton Task (maven-package step):
env:
  - name: NEXUS_USER
    valueFrom:
      secretKeyRef:
        name: cms-nexus-secret
        key: username
  - name: NEXUS_PASSWORD
    valueFrom:
      secretKeyRef:
        name: cms-nexus-secret
        key: password
⚠️
Create Secret: oc create secret generic cms-nexus-secret --from-literal=username=cms-deployer --from-literal=password=<pw> -n rbi-cms

oc get pods -n rbi-cms

# Check health endpoints:
for svc in cms-backend cms-api-gateway cms-rules-service cms-workflow-service; do
  echo "--- $svc ---"
  oc exec deploy/$svc -n rbi-cms -- wget -qO- http://localhost:8080/actuator/health
done

Pre-Deployment Checklist

Infrastructure Requirements

  • MySQL/Oracle database — accessible from the OpenShift cluster
  • Keycloak — with cms realm configured (roles, clients, users)
  • Kafka — 3-node cluster (or single-node for dev/staging)
  • OpenSearch — for full-text complaint search
  • Nexus — for hosting cms-parent and cms-common JARs
  • Persistent Storage (NFS/Ceph) — for attachment and storage PVCs
  • Container Registry — accessible from build pods

Secrets to Configure

  • cms-db-secret — DB_USER, DB_PASSWORD, SMTP credentials
  • cms-keycloak-secret — client ID, client secret, admin credentials
  • cms-sms-secret — SMS gateway API key, sender ID, template ID
  • cms-opensearch-secret — OpenSearch username and password
  • cms-kafka-secret — Kafka SASL credentials (if auth enabled)
  • cms-nexus-secret — Nexus deploy user credentials (for build pipeline)

Routes to Configure

  • Public portal: Update spec.host in deployment-portal-frontend.yaml
  • Staff portal: Update spec.host in deployment-frontend.yaml

Database

  • Schema creation: Hibernate ddl-auto: update creates tables on first startup
  • No manual migration scripts needed — JPA handles DDL
  • Seed data: DataInitializer seeds reference data automatically in non-prod profiles
