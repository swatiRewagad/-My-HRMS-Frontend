CMS 2.0 — Complete Docker Desktop Setup Guide

Pull the repo and run the entire CMS application on Docker Desktop — No MySQL, Kafka, or Redis installation needed

Quick Start Prerequisites Architecture 1. Clone & Configure 2. Docker Compose Up 3. Keycloak Setup 4. Verify 5. Dev Mode (Frontend) 6. Test Users Troubleshooting Useful Commands

⚡ Quick Start (TL;DR)

If you have Docker Desktop running with 8 GB RAM allocated:

# 1. Clone
git clone https://github.com/swatiRewagad/-My-HRMS-Frontend.git
cd My-HRMS-Frontend/CMS2.0/cms-infra

# 2. Create .env (optional - only needed for AI features)
echo "GROQ_API_KEY=gsk_your_key_here" > .env

# 3. Start everything (MySQL, Redis, Kafka, 2x Backend, PaddleOCR, Frontend, Nginx)
docker compose up -d --build

# 4. Wait ~3-4 minutes for all containers to be healthy
docker compose ps

# 5. Open browser
# App: http://localhost
# Keycloak Admin: http://localhost:8180 (admin/admin)

That's it! No MySQL, Kafka, Redis, or Java installation needed on your machine.

2 Start Everything with Docker Compose

2.1 Build & Start All Containers

# Navigate to the infrastructure folder
cd CMS2.0/cms-infra

# Build all images and start containers in detached mode
docker compose up -d --build

# First run takes 5-10 minutes (downloads base images, builds from source)
# Subsequent runs take ~30 seconds (uses cached layers)
First-time build is slow! Docker needs to:
  • Download base images (~2 GB total: Java, Node, Python, MySQL, Redis, Nginx)
  • Build backend (Maven downloads dependencies: ~500 MB)
  • Build frontend (npm install + ng build: ~400 MB)
  • Download PaddleOCR models (~1.5 GB)
Total first-run download: ~4-5 GB. After first build, everything is cached.

2.2 Watch Build Progress

# See real-time build output (useful for first run)
docker compose up --build

# Or follow logs of specific service
docker compose logs -f cms-backend-1

# Once you see "Started CmsApplication" in backend logs, it's ready

2.3 Check Container Status

# All containers should show "running (healthy)" or "running"

2.4 Startup Order (Automatic)

Docker Compose handles the startup order automatically via depends_on + health checks:

  1. MySQL starts first → waits until healthy (accepts connections)
  2. Redis starts first → waits until healthy (responds to PING)
  3. Kafka starts first → waits until healthy (broker ready, KRaft mode)
  4. Backend 1 & 2 wait for MySQL + Redis + Kafka to be healthy, then start
  5. PaddleOCR starts independently (no dependencies)
  6. Frontend starts independently
  7. Gateway (Nginx) waits for both backends + frontend
When all containers are healthy, the application is accessible at:
http://localhost — Main application (through Nginx gateway)

3 Keycloak Setup (Authentication)

Note: Keycloak is NOT included in the Docker Compose file. If you need SSO/authentication, run Keycloak separately. For testing without auth, the app works with session-based login (hardcoded users via sessionStorage).

Option A: Skip Keycloak (Quick Testing)

The CRPC module uses session-based authentication by default. You can test without Keycloak:

# Just open the app - CRPC module auto-assigns user via session
http://localhost

# Navigate to CRPC staff portal
http://localhost/crpc

Option B: Run Keycloak (Full SSO)

If you want proper Keycloak SSO, start it separately:

# Run Keycloak in Docker (separate from main compose)
docker run -d --name cms-keycloak \
  -p 8180:8080 \
  -e KEYCLOAK_ADMIN=admin \
  -e KEYCLOAK_ADMIN_PASSWORD=admin \
  quay.io/keycloak/keycloak:26.0 start-dev

# Wait 30 seconds, then access:
# http://localhost:8180

File: cms-portal-frontend/src/environments/environment.ts

export const environment = {
  production: false,
  apiBaseUrl: 'http://localhost:8082',     // Backend (direct)
  keycloakUrl: 'http://localhost:8180',    // Keycloak (if running)
  realm: 'cms',
  sessionTimeoutMinutes: 15,
  maxFileSizeMB: 5,
  maxTotalUploadSizeMB: 25,
  maxFileCount: 10,
  allowedFileExtensions: ['.pdf', '.doc', '.docx', '.jpg', '.jpeg', '.png', '.xls', '.xlsx'],
};
Note: When running frontend locally (port 4300), it connects directly to the backend on port 8082. The Docker backend exposes no host port by default — it's only reachable through Nginx (port 80). To access backend directly for dev, you can add a port mapping or use the Nginx proxy.

! Troubleshooting

Build Fails: "Cannot connect to Docker daemon"

Make sure Docker Desktop is running. Check the system tray for the Docker whale icon.
# Restart Docker Desktop from system tray → Right-click → Restart
# Or restart WSL:
wsl --shutdown
# Then reopen Docker Desktop

Build Fails: "Out of memory" or "Killed"

Docker doesn't have enough RAM allocated.
# Fix: Docker Desktop → Settings → Resources → Memory: 8 GB minimum
# Apply & Restart, then try again:
docker compose down
docker compose up -d --build

Backend Won't Start: "Connection refused to MySQL"

# Backend starts before MySQL is ready (shouldn't happen with health checks)
# Manually restart backend after MySQL is healthy:
docker compose restart cms-backend-1 cms-backend-2

# Check MySQL status:
docker compose ps cms-mysql
docker compose logs cms-mysql | tail -20

Port Already in Use

# Find what's using port 80:
netstat -ano | findstr :80
# Kill the process (replace PID):
taskkill /PID 12345 /F

# Common culprits:
# Port 80: IIS, Apache, Skype
# Port 3307: Another MySQL instance
# Port 8180: Local Keycloak

# Stop IIS if it's using port 80:
net stop w3svc

Frontend Shows Blank Page

# Check if frontend container built successfully:
docker compose logs cms-frontend

# Common issue: node_modules cache. Rebuild without cache:
docker compose build --no-cache cms-frontend
docker compose up -d cms-frontend

PaddleOCR Container Keeps Restarting

# PaddleOCR needs 2 GB RAM. Check Docker allocation:
docker stats cms-paddle-ocr

# If OOM killed, increase Docker memory to 10 GB
# Or skip OCR by commenting it out in docker-compose.yml:
# cms-paddle-ocr:
#   ... (comment entire service)

Changes Not Reflecting After Code Edit

# Docker caches build layers. Rebuild the changed service:
docker compose up -d --build cms-backend-1 cms-backend-2   # For backend changes
docker compose up -d --build cms-frontend                  # For frontend changes

# Or rebuild everything:
docker compose up -d --build

"CORS Error" in Browser Console

# The backend allows these origins by default:
# http://localhost:4200, http://localhost:4201, http://localhost:4202,
# http://localhost:4300, http://localhost

# If using a different port, set in .env:
CMS_CORS_ALLOWED_ORIGINS=http://localhost:4200,http://localhost:4300,http://localhost:5000

# Then restart backends:
docker compose restart cms-backend-1 cms-backend-2

Nuclear Option: Fresh Start

# Stop everything and DELETE all data (DB, Redis cache, volumes)
docker compose down -v

# Remove built images (forces full rebuild)
docker compose down -v --rmi all

# Clean Docker system (removes unused images/containers)
docker system prune -a --volumes

# Start fresh
docker compose up -d --build
