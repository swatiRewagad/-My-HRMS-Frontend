1. MySQL Database Setup

Why MySQL?
CMS stores all complaint data, entities, timelines, attachments metadata, and email drafts in MySQL. The cms-backend (monolith) uses MySQL directly with HikariCP connection pooling. Production uses Oracle, but dev uses MySQL for simplicity.

1.1 Installation

Option A: Install on Linux Server

# RHEL/CentOS
sudo dnf install mysql-server
sudo systemctl start mysqld
sudo systemctl enable mysqld

# Get temporary root password
sudo grep 'temporary password' /var/log/mysqld.log

# Secure installation
sudo mysql_secure_installation

Option B: Docker Container

docker run -d \
  --name cms-mysql \
  -e MYSQL_ROOT_PASSWORD=root \
  -e MYSQL_DATABASE=cms_db \
  -e MYSQL_USER=cms_user \
  -e MYSQL_PASSWORD=cms_pass \
  -p 3306:3306 \
  --restart unless-stopped \
  mysql:8.0 \
  --max-connections=500 \
  --character-set-server=utf8mb4 \
  --collation-server=utf8mb4_unicode_ci

Option C: Already Running on Dev

If MySQL is already on the dev server, just get the connection details (host, port, username, password, database name) from your DBA/admin.

1.2 Configuration

Create Database & User

mysql -u root -p

CREATE DATABASE cms_db CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;
CREATE USER 'cms_user'@'%' IDENTIFIED BY 'cms_pass';
GRANT ALL PRIVILEGES ON cms_db.* TO 'cms_user'@'%';
FLUSH PRIVILEGES;
Why '%' host?
Allows connections from any IP — needed so your laptop can connect to the dev server's MySQL.

Allow Remote Connections

Edit /etc/mysql/mysql.conf.d/mysqld.cnf (or /etc/my.cnf):

[mysqld]
bind-address = 0.0.0.0
port = 3306
max_connections = 500
innodb_buffer_pool_size = 1G
character-set-server = utf8mb4

Restart MySQL:

sudo systemctl restart mysqld

Open Firewall

sudo firewall-cmd --add-port=3306/tcp --permanent
sudo firewall-cmd --reload

1.3 Schema & Initial Data

The backend auto-creates tables on first startup (ddl-auto: update), but you can also run the schema script:

mysql -u cms_user -p cms_db < CMS2.0/cms-backend/src/main/resources/db/cms_database_scripts.sql
Auto-Initialization
The DataInitializer.java class seeds sample data (entities, categories, complaints) on first boot. No manual data load required.

2. Apache Kafka Setup

Why Kafka?
CMS uses event-driven architecture. When a complaint is filed, Kafka distributes events to multiple services:
  • complaint.ingested → Workflow starts processing, Assignment finds officer, Search indexes it
  • complaint.assigned → Notification sends email/SMS to assigned officer
  • complaint.escalated → SLA Monitor alerts supervisor
  • complaint.resolved/closed → Audit logs final state, Notification informs complainant
Without Kafka, services can't communicate asynchronously — you'd lose event-driven decoupling.

2.3 Option B: Standalone Kafka (KRaft Mode)

Use this if your dev server is a plain Linux VM without OpenShift.

Download & Install

cd /opt
wget https://downloads.apache.org/kafka/3.7.0/kafka_2.13-3.7.0.tgz
tar -xzf kafka_2.13-3.7.0.tgz
ln -s kafka_2.13-3.7.0 kafka
mkdir -p /var/kafka-data

Configure KRaft (No Zookeeper)

Edit /opt/kafka/config/kraft/server.properties:

# Node configuration
node.id=1
process.roles=broker,controller

# IMPORTANT: Replace DEV_SERVER_IP with actual IP
listeners=PLAINTEXT://0.0.0.0:9092,CONTROLLER://0.0.0.0:9093
advertised.listeners=PLAINTEXT://DEV_SERVER_IP:9092

# Controller
controller.listener.names=CONTROLLER
controller.quorum.voters=1@localhost:9093
listener.security.protocol.map=CONTROLLER:PLAINTEXT,PLAINTEXT:PLAINTEXT

# Storage
log.dirs=/var/kafka-data

# Topic defaults
num.partitions=3
default.replication.factor=1
auto.create.topics.enable=true

# Retention (7 days)
log.retention.hours=168
log.segment.bytes=1073741824

# Performance
num.network.threads=3
num.io.threads=8
socket.send.buffer.bytes=102400
socket.receive.buffer.bytes=102400
Why KRaft?
KRaft eliminates ZooKeeper — fewer moving parts, simpler setup, single process. Apache Kafka has deprecated ZooKeeper in favor of KRaft.

Initialize & Start

# Generate cluster ID (one-time)
KAFKA_CLUSTER_ID=$(/opt/kafka/bin/kafka-storage.sh random-uuid)
echo "Cluster ID: $KAFKA_CLUSTER_ID"

# Format storage (one-time)
/opt/kafka/bin/kafka-storage.sh format \
  -t $KAFKA_CLUSTER_ID \
  -c /opt/kafka/config/kraft/server.properties

# Start Kafka
/opt/kafka/bin/kafka-server-start.sh -daemon /opt/kafka/config/kraft/server.properties

# Verify it's running
/opt/kafka/bin/kafka-topics.sh --bootstrap-server localhost:9092 --list

Create systemd Service

sudo tee /etc/systemd/system/kafka.service << 'EOF'
[Unit]
Description=Apache Kafka (KRaft)
After=network.target

[Service]
Type=simple
User=kafka
Environment="KAFKA_HEAP_OPTS=-Xms512m -Xmx1g"
ExecStart=/opt/kafka/bin/kafka-server-start.sh /opt/kafka/config/kraft/server.properties
ExecStop=/opt/kafka/bin/kafka-server-stop.sh
Restart=on-failure
RestartSec=10

[Install]
WantedBy=multi-user.target
EOF

sudo systemctl daemon-reload
sudo systemctl enable kafka
sudo systemctl start kafka
Why systemd?
Ensures Kafka auto-starts on server reboot and auto-restarts on crash. Essential for a shared dev environment.

Open Firewall

sudo firewall-cmd --add-port=9092/tcp --permanent
sudo firewall-cmd --reload

2.4 Create CMS Topics

Why Pre-Create Topics?
While auto.create.topics.enable=true works, pre-creating gives you control over partition count and retention. More partitions = more parallel consumers = higher throughput.

Option A: Via Kafka UI (Easiest)

  1. Open https://my-console.apps.ocpcl.rebit.local/
  2. Navigate to Topics → Create Topic
  3. Create each topic with settings below

Option B: Via CLI (Standalone)

BOOTSTRAP=localhost:9092   # or DEV_SERVER_IP:9092

# Create all CMS topics
for topic in complaint.ingested complaint.assigned complaint.inprogress \
             complaint.escalated complaint.resolved complaint.closed; do
  /opt/kafka/bin/kafka-topics.sh --bootstrap-server $BOOTSTRAP \
    --create --topic $topic --partitions 3 --replication-factor 1 \
    --config retention.ms=604800000
done

# Dead Letter Queue (single partition, longer retention)
/opt/kafka/bin/kafka-topics.sh --bootstrap-server $BOOTSTRAP \
  --create --topic complaint.dlq --partitions 1 --replication-factor 1 \
  --config retention.ms=2592000000

3. Keycloak Setup

Why Keycloak?
CMS 2.0 uses Keycloak for:
  • Authentication: OAuth2/OIDC login — no custom auth code needed
  • Authorization: Role-based access control (11 roles for CRPC/CEPC/RBIO workflows)
  • User Management: Officers, admins stored centrally with role assignments
  • JWT Tokens: API Gateway validates tokens issued by Keycloak
  • Round-Robin: Backend queries Keycloak Admin API for users-by-role to distribute work
  • SSO: Single sign-on across all CMS modules

3.1 Installation

Option A: Standalone (Dev Server)
Recommended

Direct install on Linux. Uses PostgreSQL for persistence.

Option B: Docker
Quick

Single container with embedded H2 or external PostgreSQL.

Option C: OCP Operator
Enterprise

Red Hat SSO / Keycloak Operator on OpenShift.

3.2 Configuration

Setup PostgreSQL (Keycloak's Database)

# Install PostgreSQL
sudo dnf install postgresql-server postgresql
sudo postgresql-setup --initdb
sudo systemctl start postgresql
sudo systemctl enable postgresql

# Create database for Keycloak
sudo -u postgres psql <<SQL
CREATE USER keycloak WITH PASSWORD 'keycloak_password';
CREATE DATABASE keycloak OWNER keycloak;
GRANT ALL PRIVILEGES ON DATABASE keycloak TO keycloak;
SQL
Why PostgreSQL for Keycloak?
Keycloak stores realms, users, sessions, roles, and client configurations. H2 (embedded) is unstable under load and loses data on restart. PostgreSQL provides durability for a shared dev environment.

Configure Keycloak

Edit /opt/keycloak/conf/keycloak.conf:

# ─── Database ───
db=postgres
db-url=jdbc:postgresql://localhost:5432/keycloak
db-username=keycloak
db-password=keycloak_password

# ─── HTTP ───
http-enabled=true
http-port=8080
http-host=0.0.0.0

# ─── Hostname (IMPORTANT: use dev server IP) ───
hostname=DEV_SERVER_IP
hostname-strict=false
hostname-strict-https=false

# ─── Proxy (if behind load balancer) ───
proxy-headers=xforwarded

# ─── Logging ───
log-level=INFO
Why hostname=DEV_SERVER_IP?
JWT tokens contain an issuer URL. If this doesn't match what your backend expects, token validation fails with "Invalid issuer". The hostname must be what clients (your laptop) use to reach Keycloak.

Start Keycloak

# Dev mode (no HTTPS required, all features enabled)
/opt/keycloak/bin/kc.sh start-dev --http-port=8080

# OR as background service
nohup /opt/keycloak/bin/kc.sh start-dev --http-port=8080 > /var/log/keycloak.log 2>&1 &

Create systemd Service

sudo tee /etc/systemd/system/keycloak.service << 'EOF'
[Unit]
Description=Keycloak Identity Server
After=network.target postgresql.service

[Service]
Type=simple
User=keycloak
WorkingDirectory=/opt/keycloak
ExecStart=/opt/keycloak/bin/kc.sh start-dev --http-port=8080
Restart=on-failure
RestartSec=10
StandardOutput=journal
StandardError=journal

[Install]
WantedBy=multi-user.target
EOF

sudo systemctl daemon-reload
sudo systemctl enable keycloak
sudo systemctl start keycloak

Open Firewall

sudo firewall-cmd --add-port=8080/tcp --permanent
sudo firewall-cmd --reload

3.4 Client Registration

Why Clients?
A "client" in Keycloak represents an application that wants to authenticate users. CMS needs two clients:
  • cms-portal (public): For the Angular frontend — browser-based login
  • admin-cli (confidential): For the backend — server-to-server Keycloak Admin API calls (user lookup, role queries)

Why Backend Needs Admin API?
The backend calls GET /admin/realms/cms/users?briefRepresentation=true to fetch users by role for round-robin assignment. Without this, you can't auto-assign complaints to officers.
